# Beautiful UI/UX — Deep Dive Knowledge Base

A consolidated, practitioner-grade companion to `fundamentals.md`. Where the parent file is a *map* (vocabulary + pointers), this is the *territory*: how each tool actually works, the current (verified mid-2026) state of the ecosystem, concrete recipes/code, decision matrices, and the cross-cutting principles that separate "expensive" from "cheap."

**How to use:** Read §0 first — it corrects the things most likely to be stale. Then jump to the section you need. The back matter (§14–16) holds the decision logic, the expensive-vs-cheap rules, and the pre-ship gates that apply to *everything*.

## §0. Current-state deltas (what changed — read this first)

These are the facts most likely to differ from default assumptions as of June 2026. Internalize them; each one changes which tool you reach for.

- **GSAP is 100% free, including every former premium plugin** (ScrollTrigger, ScrollSmoother, SplitText, MorphSVG, DrawSVG, Flip, Inertia, Observer) — commercial use included — since **3.13 (Apr 30 2025)**, which also shipped the screen-reader-safe SplitText rewrite. Latest is **3.15 (Apr 2026)**: new `easeReverse` (adapts live mid-tween, deprecates `yoyoEase`), `quickTo`/`quickSetter` hot-path helpers, default duration now 0.5s. The "ScrollTrigger costs money" reasoning is dead.
- **Framer Motion → "Motion"** (`motion.dev`, package `motion`, React import `motion/react`; `motion/react-client` for RSC). Hybrid React + vanilla-JS, WAAPI-backed. **Springs default only for transforms** (x/y/scale/rotate); opacity/color default to tween/ease. **Vue is a *separate* package** (`motion-v`), not the same import. Smallest vanilla tween = `motion/mini` (~2.3KB).
- **Motion One is frozen** (folded into Motion) and **Theatre.js's public release is stale at 0.7.2** (1.0 moved to a private repo). Don't start new projects on either; for vanilla reach for `motion`/`motion/mini`, for springs `react-spring`.
- **Anime.js v4** (latest ~4.4.1) is a full ESM rewrite — no global `anime` object; import named fns (`animate`, `createTimeline`, `stagger`, `onScroll`, `svg`, `createScope`). A built-in `waapi` adapter (~3KB) covers WAAPI-only cases. v3 code does not run.
- **Lottie split into two players:** `lottie-web` (Airbnb, main-thread JS) vs **`dotlottie-web`** (Rust+WASM on ThorVG, off-main-thread via `DotLottieWorker`). **ThorVG 1.0 (Jan 2026)** made Software/WebGL2/WebGPU backends all production-ready. Note: the WASM player downloads *heavier* than lottie-web — the win is runtime smoothness/GPU, not first-load weight.
- **WebGPU is NOT ~95% — it's ~82–87% desktop / ~71% mobile** (Safari 26, Firefox on Win/Apple-Silicon; Firefox Linux/Android still in progress). Always keep a **WebGL2 fallback** + device-lost handling. For new 3D, plan **Three.js `WebGPURenderer` + TSL** (zero-config since r171, one-line swap with automatic fallback).
- **Three.js is r184; WebGLRenderer is now maintenance-only** — investment goes to WebGPU. **Babylon.js is on v9** (not 8; v9.0 Mar 2026) with 3DGS shadows, SOGS compression, triangle splatting, Frame Graph. Gaussian Splatting is now a real web option (Spark, PlayCanvas/SuperSplat, native loaders).
- **Native CSS scroll-driven animations** (`animation-timeline: scroll()/view()`) now ship in **Chromium 115+ AND Safari 26** — Firefox is still flagged, so NOT Baseline (~85%). Great zero-JS reveal/parallax as progressive enhancement.
- **View Transitions:** same-document (SPA) is Baseline (Oct 2025, incl. Firefox 133+); **cross-document (MPA) is Chromium + Safari only — NOT Firefox** (still flagged). Firefox is the no-transition fallback.
- **Lenis replaced Locomotive** (Locomotive v5 builds on it). The `@studio-freight/*` packages are **deprecated** → use `lenis` + `lenis/react` (`<ReactLenis root>`, `useLenis`). Lenis does **not** honor `prefers-reduced-motion` itself — don't init (or destroy) under reduced motion.
- **CSS scroll-state container queries** (`@container scroll-state(stuck/snapped/scrollable)`) style sticky/snapped elements with zero JS — Chrome 133+ only, IO fallback elsewhere.
- **Modern CSS is now Baseline:** `@property` (animatable typed custom props), `light-dark()`, `color-mix()` (Widely Available Nov 2025), `@scope`, `@starting-style` + `allow-discrete` (CSS entry/exit, often replacing `<AnimatePresence>`), and **`contrast-color()`** (Apr 2026, returns black/white). **Subgrid, container queries, `svh`/`lvh`/`dvh`** are all production-safe.
- **Anchor positioning is Baseline (early 2026)** (Chrome 125+, Safari 26+, Firefox 147+) — `anchor()`/`position-area`/`@position-try` + Popover API kills most JS positioning libs for tooltips/menus/popovers.
- **Color: OKLCH is the perceptual standard.** **APCA was *removed* from the WCAG 3 working draft** (method TBD; WCAG 3 not a Rec before ~2028) — *design* with APCA, but **ship conformance to WCAG 2.2 AA** (the operative legal benchmark for ADA/EAA).
- **INP replaced FID** (Mar 2024) and is the most-failed CWV. Targets (p75, field): **LCP ≤ 2.5s, CLS ≤ 0.1, INP ≤ 200ms** (the rumored 2.0s LCP tightening did not happen). `scheduler.yield()` helps but is **not in Safari**.
- **Tailwind is v4.3** (CSS-first `@theme`, Rust "Oxide" engine, OKLCH tokens); v4.3 adds text-shadow, masks, scrollbar/`@container-size` utilities and new palettes. **React 19.2** ships `useOptimistic` + `Activity` + `useEffectEvent`.
- **shadcn/ui CLI is v4** (multi-framework scaffolding, MCP server, namespaced registries); **Radix now ships as one unified `radix-ui` package**, and **Base UI (MUI team) hit v1.0 (Dec 11 2025, ~35 components)** under `@base-ui/react` — a serious headless default. **Tremor was acquired by Vercel and is now free/MIT.**

## §1. JavaScript Animation Libraries

### GSAP (GreenSock) — the industry standard
- **Status:** latest **3.15.0 (Apr 13, 2026)**, ~monthly cadence. The watershed was **3.13 (Apr 30, 2025)**: Webflow (which acquired GreenSock Oct 2024) made GSAP **100% free incl. every former Club plugin, commercial use included** — there is no longer any paid tier, so drop all "ScrollTrigger/SplitText cost money" reasoning. There's now an official `greensock/gsap-skills` repo of AI-agent skills for canonical patterns.
- **Model:** a property-tweening engine on a single rAF ticker (`gsap.ticker`) that interpolates *anything numeric* (CSS, SVG attrs, canvas props, JS objects, three.js values), writing inline styles. Two primitives: **Tween** (`gsap.to/from/fromTo/set(targets, vars)`) and **Timeline** (sequenced container; position params `"+=0.5"`, `"<"`/`">"` (prev start/end), `"<0.2"`, `"label"`, absolute seconds; nestable + seekable/reversible/time-scalable). `gsap.quickTo()`/`quickSetter()` are the hot-path helpers for `mousemove`/cursor work (skip per-call tween overhead).
- **Easing is its superpower:** `power1..4`, `back`, `elastic`, `expo`, `circ`, `sine`, `bounce`, `steps()`, each `.in/.out/.inOut`; `CustomEase`/`CustomBounce`/`CustomWiggle` for arbitrary curves. Default ease `power1.out`; default duration `0.5s`. New in 3.15: **`easeReverse`** lets a tween use a *different* ease when the playhead runs backward — it is recalculated from the reversal point and adapts live even on a mid-tween direction change, and works in deeply nested timelines; **`yoyoEase` is deprecated in its favor** (3.15 maps it to `easeReverse` internally, fully backwards-compatible).
- **Key plugins (all free now):** **ScrollTrigger** (`trigger`, `start`/`end`, `scrub: true|<num>`, `pin`, `snap`, `toggleActions`, `markers`, `invalidateOnRefresh`, `onUpdate(self)`); **`ScrollTrigger.batch()`** is the efficient way to reveal many elements with one shared observer (`interval`, `batchMax`, `onEnter`/`onLeave` with stagger) instead of one trigger per element; **ScrollSmoother** (smooth scroll + `data-speed`/`data-lag` parallax; requires `#smooth-wrapper > #smooth-content`; built *on* ScrollTrigger, so don't also run Lenis); **Observer** (unified wheel/touch/pointer normalization — the basis for "one-scroll-per-section" UIs without hijacking math); **Flip** (capture `Flip.getState`, mutate DOM, `Flip.from` animates the delta — the best shared-element/layout tool for vanilla); **MorphSVG**, **DrawSVG**, **MotionPathPlugin**, **Draggable**+**InertiaPlugin**, **ScrambleText**, **Physics2D/PhysicsProps**, **SplitText**.
  - **SplitText** got a ground-up rewrite in **3.13** (~50% smaller, 14 new features): splits to `chars`/`words`/`lines`; **`autoSplit:true`** re-splits on a `ResizeObserver` width change (debounced ~200ms) and on `document.fonts` `loadingdone` so reveals survive font swaps/reflow; **`onSplit(self)`** returns the tween to keep cleanup correct; built-in **`mask:"lines"`** for clip-reveals; a11y baked in (adds `aria-label` to the parent, `aria-hidden` to the generated fragments) so screen readers still read the original string.
- **React:** `useGSAP(callback, { scope, dependencies, revertOnUpdate })` from `@gsap/react` — wraps in `gsap.context()` and **auto-reverts on unmount** (kills the #1 React-GSAP footgun: leaked tweens/ScrollTriggers). `scope` (a ref) confines selector text to that subtree; **`contextSafe()`** wraps any animation created *after* the initial run (event handlers) so it's tracked + reverted too. Register plugins at module scope.
- **Responsive + reduced motion:** `gsap.matchMedia()` — `mm.add("(min-width:800px)", () => {...})` and `mm.add("(prefers-reduced-motion: reduce)", () => {...})`; the setup function's tweens/ScrollTriggers auto-revert when the query stops matching. This is the idiomatic reduced-motion gate for GSAP.
- **Gotchas:** animate `x/y/scale/rotation/opacity`, never `top/left/width/margin` (layout thrash); register plugins once (`gsap.registerPlugin(ScrollTrigger, useGSAP)`); call `ScrollTrigger.refresh()` after async layout/image loads (or set `invalidateOnRefresh:true` so functional values recompute); core is ~**52KB min** but plugins are per-file/tree-shakeable. SSR: guard `window`/register inside `useGSAP`/`useEffect`.

```js
gsap.registerPlugin(ScrollTrigger);
gsap.timeline({ scrollTrigger:{ trigger:".panel", start:"top center", end:"+=500", scrub:true, pin:true }})
  .to(".box", { x:300, rotation:360, ease:"none" })
  .to(".box", { backgroundColor:"var(--accent)" }, "<");
```

### Motion (formerly Framer Motion)
- **Status:** latest **12.x (12.40.0, mid-2026)**. Home `motion.dev`, canonical package `motion`, React import `motion/react` (use `motion/react-client` to mark components as client in React Server Component trees). Now a **hybrid React + vanilla-JS + Vue** library; `framer-motion` still ships as an alias but is frozen — migrate imports. A paid **Motion+** tier exists for premium extras (e.g. `Cursor`, splitting helpers, Motion Studio), but the core library is free/MIT. **Vue is a separate package** (`motion-v`, ~v2.3.0, `import { ... } from "motion-v"`) — not the same `motion` import as React.
- **Model (React):** `<motion.div>` driven by `initial`/`animate`/`exit` (can be **variants** for orchestration with `staggerChildren`/`delayChildren`/`when`), `transition` (`type:"spring"` with `stiffness`/`damping`/`mass` *or* the friendlier `bounce`+`duration`; or `tween` with `duration`/`ease`), gesture props `whileHover`/`whileTap`/`whileFocus`/`whileInView`/`whileDrag`, and `drag`+`dragConstraints`+`dragMomentum`.
- **Springs are the default — but selectively:** physical/transform props (`x`, `y`, `scale`, `rotate`) animate with **spring** physics (interruptible, velocity-carrying); non-physical props (`opacity`, `color`, `backgroundColor`) default to **tween/ease** because a bouncy opacity reads wrong. Override per-property in `transition`.
- **`<AnimatePresence>`** enables `exit` animations (defers unmount until they finish); modes `wait` (out-then-in), `popLayout` (exiting element popped from flow so siblings reflow immediately), `sync`. Children need stable `key`s.
- **Layout animations:** `layout` prop auto-animates size/position changes (FLIP under the hood, so it transforms — cheap); **`layoutId`** does shared-element "magic move" across mount/unmount; `<Reorder.Group>`/`<Reorder.Item>` for drag-to-reorder lists. Watch for distortion on `border-radius`/`box-shadow` (animate via `style`, not `layout`).
- **Hooks (motion values, off-React-render):** `useScroll` (`scrollYProgress`), `useTransform`, `useSpring`, `useMotionValue`, `useMotionValueEvent`, `useInView`, `useAnimate` (imperative scoped `animate` + timeline sequences).
- **Hybrid engine + bundle control:** runs on **WAAPI/ScrollTimeline** (hardware-accelerated, off main thread, ~120fps) for independent transforms/opacity; falls back to the JS engine for springs, layout, interruption, and orchestration. Sizes that matter: the all-in `motion` React component ≈ **34KB**; swap to **`LazyMotion` + the slim `m` component** (load `domAnimation` ≈ 6KB initial, or `domMax` for drag/layout) to cut the initial bundle hard. For the smallest possible vanilla footprint, **`motion/mini`** `animate` is **~2.3KB** (pure WAAPI, the smallest animation tool for React) vs the hybrid `animate` from `motion` at ~17KB; the standalone `scroll()` function is ~5.2KB.
- **Motion+** (`motion-plus` / `motion-plus-react`, paid one-time license; actively dev'd 2026) — premium add-on to Motion with ready-made, hard-to-build primitives: **Cursor** (magnetic / zoning / morph-to-target follower), **Ticker** (infinite marquee that fixes the CSS-marquee gotchas), **Carousel** (wheel + drag), **AnimateNumber** (digit spin, `trend` prop), plus **Motion Studio** (visual/in-editor authoring) and premium examples. The buy-vs-build option for magnetic cursors (§4), marquees (§4), and animated numbers if you're already on Motion.

```jsx
import { motion, AnimatePresence } from "motion/react";
<AnimatePresence mode="wait">
  {open && <motion.div layoutId="card"
    initial={{opacity:0,y:20}} animate={{opacity:1,y:0}} exit={{opacity:0,y:-20}}
    transition={{type:"spring",stiffness:300,damping:30}} whileHover={{scale:1.05}} />}
</AnimatePresence>
```

### The rest
- **Motion One** — the original standalone WAAPI micro-lib. **Now folded into Motion's vanilla surface**; its repo is essentially frozen (last meaningful commits 2024). Don't start new projects on it — reach for `motion` / `motion/mini` instead (same WAAPI core, maintained). Trade-off of the pure-WAAPI path either way: springs are approximated, and there's no FLIP/`layout`/shared-element.
- **Anime.js v4** — free MIT, lightweight, tree-shakeable. Latest **4.4.1 (~Apr/May 2026)**; v4 is a full ESM rewrite (the global `anime` object is **gone**, v3 code does not run). The sweet spot between Motion One and GSAP for vanilla timelines. Import named: `import { animate, createTimeline, stagger, utils, onScroll, createDraggable, createScope, svg, spring, eases, waapi, splitText, createLayout, engine } from "animejs"`. Notables: **`onScroll()`** is a full ScrollObserver (thresholds, sync modes incl. smooth-scroll) so you don't always need GSAP for scroll; **`svg`** namespace gives `morphTo`/`createMotionPath`/`createDrawable`; a built-in **`waapi`** adapter renders a WAAPI-only animation at ~3KB (vs ~10KB full engine); `splitText` + `createLayout` cover text and FLIP. `then()`/`utils.promisify` for promises; **`createScope({ root })`** is the React/cleanup primitive (auto-reverts, supports `mediaQueries`).
- **react-spring** (v10; `@react-spring/web`) — spring-physics animation for React from the pmndrs orbit; a lighter, hooks-first alternative to Motion when you want pure springs and tight R3F interop (shares ecosystem with R3F/Drei). Less full-featured than Motion for layout/exit/variants — reach for it when springs are the whole story, Motion when you need orchestration/shared-element.
- **Lottie** — plays After-Effects vector animations as JSON. Two players today: **(1)** `lottie-web` (Airbnb, latest **5.13.0**, still maintained) with `svg` (fidelity, DOM-heavy), `canvas` (faster, no resolution-independence), `html` renderers — mature but JS-rasterized and main-thread-bound. **(2) Prefer `@lottiefiles/dotlottie-web`** (latest ~**0.72.x**): a **Rust+WASM core (`dotlottie-rs`) powered by ThorVG**, which hit **1.0 on Jan 31 2026** with **Software (Canvas2D), WebGL2, and WebGPU** backends now *all production-ready* (GPU >150% faster than ThorVG 0.15) plus full **dotLottie v2** support. **`DotLottieWorker`** renders on a Web Worker via `OffscreenCanvas` — keeps the main thread free with dozens of animations. The `.lottie` container is a zipped bundle (smaller transfer, multi-animation + theming), but note the **WASM player itself is heavier to download than lottie-web's JS** — the win is runtime smoothness/quality, not first-load weight. This is timeline *playback*, not data-bound interactive logic.
- **Rive** — real-time interactive vector runtime + editor; React runtime `@rive-app/react-canvas` (~**4.28.0**). **State machines + data binding live inside the `.riv` file** → genuine interactive/stateful/data-driven graphics with tiny payloads. React: `useRive` + `useStateMachineInput`; for data binding, drive a **ViewModel instance** (`useViewModel`/`useViewModelInstance`) so the editor's bindable props map to your app state declaratively (often replacing manual state-machine inputs). Renderer choice matters: **`@rive-app/webgl2` (the high-fidelity "Rive Renderer", default for new work)** vs **`@rive-app/canvas` (Canvas2D — use for grids/lists since it has no WebGL context limit)**; for many instances set `useOffscreenRenderer:true` to share one offscreen GL context. Pick Rive over Lottie for interactive UI characters, game-like UI, and data-driven motion.
- **Theatre.js** — visual sequencer + studio for keyframed 3D/DOM motion (great with R3F via `@theatre/r3f`). **Caveat:** the public npm packages (`@theatre/core`/`studio`) are still **0.7.2 — no release since May 2024** (~2 years) — because the team moved 1.0 development into a **private repo to iterate faster** (it is *not* abandoned, and a 1.0 is promised). For now treat it as **dormant / pre-1.0 with a stale public release**: fine for internal/portfolio tooling, risky as a production dependency until 1.0 actually ships.
- **Auto-Animate** (`@formkit/auto-animate`, ~**0.9.0**) — one-line FLIP for add/remove/move of a parent's *direct children*; ~2KB, respects `prefers-reduced-motion` out of the box, framework adapters for React/Vue/Solid. Zero-config wins for lists/accordions/grids; outgrow it (sequencing, exit choreography, shared elements) → Motion `layout` / GSAP Flip.
- **Popmotion / Velocity** — both **unmaintained legacy**. Velocity has had no meaningful commits in years; Popmotion isn't formally archived but its primitives were absorbed into Motion (its direct lineage) and it no longer ships updates. Migrate off — Motion is the supported successor.

**Verified stamps (npm, mid-2026):** core animation versions to pin — Motion **12.40**, Anime.js **4.4**, Lenis **1.3** (*import from the bare **`lenis`** package; the legacy `@studio-freight/lenis` is frozen at 1.0.42, migrate it*), Auto-Animate **0.9**, react-spring **10.x**. (R3F/Drei/Three/Pixi/p5 version stamps live in §2.)

**Decision:** scroll storytelling / SVG morph / draw / text-split → **GSAP** (+ ScrollTrigger/SplitText, all free); declarative React + gestures + enter/exit + layout/shared-element + springs → **Motion**; pure-springs React with R3F interop → **react-spring**; smallest WAAPI tween (vanilla) → **`motion/mini`** (~2.3KB); free lightweight general timeline with built-in scroll → **Anime.js v4**; designer-authored AE motion → **Lottie/dotLottie** (dotLottie for GPU + worker); interactive/stateful/data-bound graphics → **Rive**; effortless list/accordion motion → **Auto-Animate**; already on Motion and want a magnetic cursor/marquee/animated number off the shelf → **Motion+**. Avoid starting on **Motion One / Popmotion / Velocity** (frozen) and weigh **Theatre.js** carefully (no public release since May 2024, 1.0 still private).

## §2. 3D / WebGL / Generative Graphics

**Big picture:** Two GPU APIs — WebGL2 (universal, ~98%) and WebGPU (now broadly shipping, adds compute shaders + lower CPU/driver overhead + WGSL). 2025–26 is the crossover: libraries dual-target with a **WebGPU→WebGL2 fallback**, and Three.js has officially shifted its roadmap so **`WebGLRenderer` is maintenance-only** (still supported, no major new features) while `WebGPURenderer` gets the investment. Abstraction ladder: Spline (no-code) → R3F+Drei / Babylon (batteries) → Three.js (engine) → OGL (thin) → raw WebGL/WebGPU. **Node-based shading** (Three's TSL, Babylon NodeMaterial, p5.strands) is the dominant new paradigm because one graph compiles to *both* GLSL and WGSL — write once, run on either backend.

- **Three.js** (**r184**, Apr 16 2026; ~monthly cadence) — retained-mode scene graph: `Scene` → `Mesh` (`BufferGeometry` + `Material`) + `Camera` + `Renderer`. Material tiers: Basic → Lambert/Phong → **Standard/Physical (PBR)** → ShaderMaterial → **`*NodeMaterial` (TSL)**. Two renderers coexist: `WebGLRenderer` (stable, GLSL, **maintenance-mode** — no big new features) and **`WebGPURenderer`** (the strategic future). Since **r171 (Sep 2025)** the WebGPU path is **zero-config**: `import * as THREE from 'three/webgpu'` (or `import { WebGPURenderer } from 'three/webgpu'`), `await renderer.init()`, and it **auto-falls-back to a WebGL2 backend** on unsupported browsers — a true one-line swap. Addons live in `three/addons/` (OrbitControls, GLTFLoader, EffectComposer) — import per-file. New WebGPU post-processing stack has built-in MRT + automatic pass-combining via node materials.
  - **Perf (hard numbers):** keep **draw calls < 100** (>500 degrades even on strong GPUs) — watch `renderer.info.render.calls`; collapse them via `InstancedMesh` (same geo) / **`BatchedMesh`** (r156+, *different* geos sharing one material) / merged geometry / reuse materials. **Dispose manually** (`geometry/material/texture.dispose()`; GLTF `ImageBitmap` textures also need `texture.source.data.close?.()`) — GPU resources never GC, so leaks are silent; watch `renderer.info.memory` for climbing counts. Clamp `setPixelRatio(Math.min(devicePixelRatio, 2))`. **KTX2/Basis cuts VRAM ~10×** (a 200KB PNG ≈ 20MB uncompressed in GPU memory) — UASTC for hero/normal maps, ETC1S for size; DRACO/Meshopt for geometry. Use `renderer.setAnimationLoop()` (required for WebGPU/XR, not raw `requestAnimationFrame`). Core is **~155KB gzip** and **tree-shakes poorly** — budget for it.

```js
// WebGPU + TSL, with automatic WebGL2 fallback
import * as THREE from 'three/webgpu';
import { color, time, oscSine, positionLocal } from 'three/tsl';
const renderer = new THREE.WebGPURenderer({ antialias: true });
await renderer.init();                       // async — WebGPU needs device acquisition
const mat = new THREE.MeshStandardNodeMaterial();
mat.colorNode = color(0xff0044).mul(oscSine(time));   // animated, compiles to GLSL+WGSL
renderer.setAnimationLoop(() => renderer.render(scene, camera));
```

- **React Three Fiber (R3F)** (**v9.x** stable for React 19, compatible across 19.0–19.2 incl. the Activity feature; **v10 is alpha/canary only — not yet beta, don't ship it** — new scheduler + data-oriented workflow) — a React reconciler rendering to the Three scene graph. JSX maps to Three classes (`<mesh>`, `<boxGeometry args={[1,1,1]}>`), props set properties, `attach` wires child→parent property. `useFrame((state,dt)=>...)` is the render loop — **mutate refs, never `setState` per frame** (that re-renders React every frame). `frameloop="demand"` (re-render only on prop/control change) for static/hero scenes — big battery + INP win. Pairs with the **`three/webgpu`** entry for WebGPU.
- **Drei** (**~150** R3F helpers) — load-bearing: `<OrbitControls>`, **`<Environment>`** (HDRI/IBL — the single thing that makes PBR/glass look right), `useGLTF`/`useTexture` (with `useGLTF.preload`; `useGLTF` has DRACO + Meshopt decode **on by default** — decoder from a Google CDN, pass `useGLTF(url, "/draco/")` for self-hosted/offline), **`<Html>`** (DOM in 3D — the accessible-content escape hatch), **`<Text>`** (SDF/troika, crisp at any scale), `shaderMaterial()` factory (typed uniforms + auto `extend`), **`<MeshTransmissionMaterial>`** (glass/refraction — gorgeous but expensive, keep samples low), `<Instances>`/`<Merged>` (draw-call reduction), `<PerformanceMonitor>`/`<AdaptiveDpr>`/`<Detailed>` (LOD). Drei also has growing TSL/WebGPU-aware helpers.
- **Physics & post (R3F add-ons):** **`@react-three/rapier`** (Rapier, Rust→WASM) is the current physics standard — fast, **deterministic** (same inputs → same sim across devices), v2 supports R3F v9/React 19; the older `@react-three/cannon` (use-cannon) is effectively superseded. **`@react-three/postprocessing`** wires the `postprocessing` effect composer (bloom, DoF, SSAO, vignette) into R3F declaratively.
- **uikit** (`@react-three/uikit` / `@pmndrs/uikit`, v1 2026) — render real **flexbox UI inside WebGL** (buttons, text, inputs, nested scroll, theming, light/dark) via Yoga; instanced + fast, runs on web/AR/VR/native. Use for in-canvas HUDs and **spatial/VR menus** that must live in the GL layer (vs Drei `<Html>`, which is a DOM overlay, not GL-native or VR-capable). Supersedes the abandoned `react-three-flex`.
- **Babylon.js 9** (**v9.0 Mar 26 2026**; 9.12+) — full batteries-included engine (Havok physics, character controller, GUI, **Node Material Editor**, all-new **Inspector**, spatial audio, **native Gaussian splats** w/ SOG/SPZ). v9 adds **Clustered Lighting** (hundreds–thousands of lights on WebGPU *and* WebGL2), **Volumetric Lighting**, **Frame Graph v1** (fine-grained render-pipeline control), **Node Particle Editor**, animation retargeting, triangle splatting + composite splat scenes with global sorting, and geospatial rendering. v8+ ships every shader in **GLSL *and* WGSL** → ~2× smaller WebGPU bundles and direct WebGPU with no conversion layer. Best-in-class **Gaussian Splatting / radiance fields** pipeline (loads `.splat`/`.ply`/`.spz`/`.sog(s)`). Pick for games/sims/integrated physics + editor, or any splat-heavy scene.
- **Gaussian Splatting (3DGS)** — photoreal radiance-field capture rendered live in-browser; the photoreal sibling of the faux-3D trend. Renderers: **Spark** (`@sparkjsdev/spark`, Three.js — fuses splats *and* mesh, 98%+ WebGL2, reads `.PLY/.SPZ/.SPLAT/.SOG`), mkkellogg **GaussianSplats3D**, **gsplat**; **PlayCanvas** + **SuperSplat** (browser editor); Three.js and **Babylon 9** support splats natively. Files are heavy (tens of MB) → lazy-load, LOD, and provide a poster fallback. The emerging move: scroll-driven splat landing pages.
- **OGL** (**v1.0.x**, e.g. 1.0.11) — thinnest WebGL2 wrapper (Three-like API, *you* write the GLSL). Tiny bundle, ideal for shader-centric hover/distortion work. Now past its long alpha but still WebGL2-only: **no WebGPU, no glTF/PBR/loaders** — bring your own.
- **PixiJS** (**v8.16.x / 8.19**, 2026) — fastest **2D** GPU renderer; WebGPU is core but **`autoDetectRenderer` defaults to WebGL2 since v8.1.0** (WebGPU was inconsistent cross-browser — opt in explicitly with `preference:'webgpu'`). Async `await app.init()`. v8.16 adds an experimental **Canvas2D fallback renderer** (runs where no GPU context exists). Games, particle systems, data-viz at scale, image filters. Atlases + batching keep draw calls low.
- **Spline** — no-code 3D (+ AI prompt-to-scene); exports a React component (`@splinetool/react-spline` + `@splinetool/runtime` peer dep), vanilla runtime, or glTF/USDZ. Fast for designers and `renderResolution`/render-on-demand by default, but the runtime + scene are **heavy multi-MB payloads (LCP-hostile)** — `lazy()` + `<Suspense>`, gate behind interaction, or export glTF and drive it yourself for control.
- **GLSL shaders** — **vertex** (per-vertex; transforms/displaces geometry) + **fragment** (per-pixel; ~99% of visual effects). Data flow: attributes → vertex; uniforms (time/resolution/textures/pointer) → both; varyings (interpolated) vertex→fragment. Canonical effects: gradients (`mix`), value/Perlin/simplex **noise + fBm** (organic motion/clouds), distortion (offset UVs by noise), dithering (Bayer/blue-noise), **fresnel** (`pow(1.0 - dot(viewDir, normal), p)` → rim/glow). Attach in Three via `ShaderMaterial`, `onBeforeCompile` (patch the built-in PBR material — keeps lighting/shadows), Drei `shaderMaterial()`, or **TSL**.
- **TSL (Three Shading Language)** — write shaders as **JS node graphs** that compile to **GLSL *and* WGSL** automatically (`import { float, vec3, uniform, texture, uv, Fn, If, Loop, time } from 'three/tsl'`). The only write-once path for both backends; assign to `material.colorNode`/`positionNode`/`emissiveNode`/etc. **Unlocks GPU compute** (particles/physics/boids) on the WebGPU backend via `instancedArray`, `instanceIndex`, `Fn(...)().compute(count)` and `renderer.compute(fn)`. The future of Three shading — learn this over hand-written GLSL for new work.

```js
// GPU compute particles (WebGPU backend) — 50k particles, no CPU loop
import { Fn, instancedArray, instanceIndex, uniform, time } from 'three/tsl';
const positions = instancedArray(50000, 'vec3');
const dt = uniform(0);
const update = Fn(() => { const p = positions.element(instanceIndex); p.addAssign(/* velocity */ p.mul(dt)); })().compute(50000);
const mat = new THREE.PointsNodeMaterial(); mat.positionNode = positions.element(instanceIndex);
// in loop: dt.value = clock.getDelta(); renderer.compute(update); renderer.render(scene, camera);
```

- **WebGPU** — successor to WebGL: compute shaders, lower CPU/driver overhead, WGSL, render bundles. **Reached Baseline (Newly Available, Jan 2026).** Support (mid-2026): Chrome/Edge (since v113), **Firefox 141 on Windows / 145 on Apple-Silicon macOS** (Linux + Android still in progress through 2026), **Safari 26** (macOS Tahoe 26 / iOS 26 / iPadOS 26 / visionOS 26). Real coverage ≈ **87% desktop / ~71% mobile** (older iOS Safari the laggard) → a **WebGL2 fallback is still mandatory, not optional**. Init is **async** everywhere; handle **device-lost** (`device.lost`) and re-create the context. For new 3D work, plan **Three.js `WebGPURenderer` + TSL** (`import 'three/webgpu'`, `await renderer.init()`, auto-falls back to WebGL2).
- **p5.js** (**v2.2.x / 2.3**) — creative coding; `createCanvas(w, h, WEBGL)`, **p5.strands** (write shaders in JS, now with branching/loops), and a **WebGPU renderer that shipped in 2.2** (p5.strands + WebGPU = compute shaders). Great for art/teaching/generative sketches, not app UIs. **2.0 is a breaking rewrite vs 1.x** (don't assume 1.x tutorials apply).
- **Two.js** (active, **v0.8.x**) — renderer-agnostic 2D vector (SVG/Canvas/WebGL), ES-module + TS types. **Paper.js** (**v0.12.17**, 2022 — long quiet but **new maintainers now pushing toward 1.0**) — still the best 2D **path geometry / boolean ops** in the browser. Pick Paper.js for vector math, Two.js for animated 2D drawing.
- **gpu-curtains** (**WebGPU, active**, v0.16.x ~Apr 2026) / **curtains.js** (WebGL, **legacy/unmaintained**) — map DOM `<img>`/`<video>` onto GPU planes that auto-sync position/size on scroll+resize for hover/scroll distortion. gpu-curtains began as a WebGPU port of curtains.js but is now a **full 3D engine** (lights, shadows, glTF, raycasting, deferred rendering, compute, instancing) — use it for DOM↔3D sync on the WebGPU stack; reach for OGL if you need the WebGL2 fallback today.

**Verified stamps (npm, mid-2026):** R3F **9.6** (`latest` = stable v9 for React 19; **v10 is alpha/canary only — not yet beta**), Drei **10.7**, Three.js **r184 / 0.184**, Babylon.js **9.12+**, PixiJS **8.19**, p5.js **2.3**.

**The honest cost of WebGL/WebGPU heroes:** big JS engine (Three core **~155KB gzip**, tree-shakes poorly) + multi-MB assets (geometry, textures, HDRIs, splats) + shader compile stalls (first-frame jank) + continuous rAF/`setAnimationLoop` → real damage to **LCP, INP, main-thread, battery, and thermals** (mobile especially). Canvas/WebGL is **opaque to assistive tech** (not focusable, not in the a11y tree, not screen-reader-readable, not find-in-page). **Hard rules:** never put essential content or navigation *only* in WebGL; ship a **DOM equivalent** (Drei `<Html>`/real markup) + a **static poster fallback** (capture a frame as an image for the no-WebGPU / low-power / `prefers-reduced-motion` / save-data path) + **lazy-load + `frameloop="demand"`** for hero scenes. Justified when the experience *is* the product (immersive site, product configurator, game, data-viz that needs the GPU); a **net negative** as a decorative background on a content/marketing site — use a shader-on-a-quad gradient, a faux-3D image sequence, or CSS instead.

## §3. Visual Styles & Aesthetic Keywords

The meta-rule for every style: **execute the underlying system, not the surface.** AI-slop is always "the look without the discipline." For each: signature → recipe → avoid-slop. Most modern recipes lean on three Baseline-safe primitives — `backdrop-filter`, `mix-blend-mode`, and grainy gradients — so verify those first.

- **Glassmorphism** — frosted translucent panels with a light edge rim that read depth via *layering*, not shadow. `background:rgba(255,255,255,.10); backdrop-filter:blur(12px) saturate(160%); -webkit-backdrop-filter:blur(12px) saturate(160%); border:1px solid rgba(255,255,255,.18)`. Keep blur **8–15px**; needs a *colorful/busy* backdrop or it's just a grey box; **`backdrop-filter` not `filter`**; never inside an `overflow:hidden`/`overflow:clip` ancestor (it clips the blur). `backdrop-filter` is Baseline *newly available* (since Sept 2024; *widely available* target Mar 16 2027) — keep the `-webkit-` prefix for older Safari; it's **GPU-cheap to declare but expensive to overuse** (each blurred layer is a full-screen pass → jank on scroll). Legibility: text over glass needs a tint/scrim or a text-shadow — WCAG contrast is measured against the *composited* result, not the panel color. Slop: frosted-purple card on a flat dark bg with no edge light, blur on everything.
- **Neumorphism / Soft UI** — same-color extrusion via **dual shadow** (`box-shadow: 10px 10px 20px #bebebe, -10px -10px 20px #fff`); `inset` = pressed/recessed. **Inherently fails WCAG (no 3:1 boundary)** — the only fix is a real border/fill, at which point it stops being neumorphism. A *dated* tell on its own (peaked ~2020); use as a decorative accent on one control, never for primary affordances or anything keyboard-reachable.
- **Skeuomorphism (modern) / Liquid Glass** — material realism: glass that *refracts* its backdrop, specular edge highlights, depth, and **content-aware legibility**. The reference is Apple **Liquid Glass** (iOS/iPadOS/macOS 26, unveiled WWDC Jun 9 2025; **mandatory for App Store apps Sept 2026**). WWDC 2026 (Jun 8) shipped a **transparency slider** (opaque ↔ clear), refraction that continues *under* full-edge sidebars, colored sidebar icons, and extra glass layers baked into icon art — i.e. Apple dialed *back* the most aggressive transparency after legibility complaints. Web approximation: glass + inset specular rims + SVG **`feDisplacementMap`** (driven by `feTurbulence`/a gradient `feImage`) for the refraction warp + `feGaussianBlur`, plus a faint parallax. **Honest limit:** CSS `backdrop-filter` *cannot* refract/bend the backdrop — true liquid-glass refraction is SVG-filter or WebGL only; ship a flat translucent fallback. Slop: photoreal leather/wood textures everywhere (the 2013-era dead style) OR a flat translucent box with no edge light masquerading as "glass."
- **Brutalism / Neo-brutalism** — hard borders + **zero-blur offset shadow** (`border:3px solid #000; box-shadow:6px 6px 0 #000; border-radius:0`), flat saturated/pastel fills, heavy grotesque type, visible structure. Examples: Gumroad, Bloomberg Businessweek, Figma marketing moments. The offset-shadow is the signature; the discipline is *typographic confidence + real content*, not the border-kit. Slop: the `6px 6px 0` border-shadow recipe sprayed onto generic SaaS cards with nothing to say.
- **Flat / Material / Material You** — **Material 3 Expressive** is the current line (announced Google I/O May 13 2025; shipped on Pixel/Android 16 QPR1 Sept 2025). Dynamic color: one (or two) seed color(s) → **HCT** (hue/chroma/tone) → 5 key colors → tonal palettes (standard tones 0–100 *plus* extended surface tones **4/6/12/17/22** dark, **87/92/94/96/97** light) → role tokens. **Tonal elevation** (surface tint) over heavy shadows; **expressive shape-morph** (rounded-square → squircle → star) and spring **motion physics** on press; emphasized type roles; **48dp** min targets. Tooling: `material-color-utilities`. Slop: generic indigo + giant drop shadows + default Roboto with none of the tonal/role system.
- **Claymorphism** — puffy 3D toy surfaces: big rounded corners + **tinted** outer shadow + two insets (`box-shadow: 0 35px 68px rgba(tint,.42), inset 0 -8px 16px rgba(tint,.2), inset 0 8px 16px rgba(255,255,255,.6)`). Pastels, 2–3 colors, generous radius (~32px+), used sparingly. The inner light-from-top + dark-from-bottom inset pair is what sells the inflation. Slop: pure-black shadows (kills the candy feel), clay balloons on every element.
- **Aurora / Mesh gradients** — overlapping soft color blobs that read like light. **CSS:** stack **4–6** `radial-gradient()` spotlights (8+ taxes low-end GPUs) on a wrapper, optionally `blur(60–100px)`; interpolate in **`linear-gradient(in oklch, …)`** (≈93% support since mid-2023) to avoid grey dead mid-tones and stepping. **Animated/brand-level:** real-time **WebGL** — this is **Stripe's actual hero gradient** (`stripe-gradient.js`: GLSL fragment shader, 3-octave simplex/fBm noise + sinusoidal mesh-warp of the UVs, ~60fps mobile). The "blurred SVG `<circle>`s + `feBlend`" trick is a fine *static* alternative, not Stripe's method. **Always add grain** (see below) to kill 8-bit banding. Slop: a two-stop *linear* gradient called a "mesh," one lazy purple→pink blob.
- **Bento grids** — modular rounded tiles of *different* sizes with one hero cell, each tile a self-contained mini-story. CSS Grid with `grid-column/row: span N`; never leave empty cells; consistent radius (~16–20px) + gap (~16px); `#f5f5f7`/`#1a1a1a`-ish surfaces. Still current (Apple keynotes, AI-product landing pages) but now a recognizable trope — earn it with content density. Slop: uniform equal boxes (destroys the hierarchy that defines bento), cards-nested-in-cards.
- **Bauhaus / Swiss / International Typographic Style** — rigorous grid, flush-left grotesque on a baseline grid, asymmetric balance, primary shapes, black/white + one accent. `grid-template-columns:repeat(12,1fr)` + spacing on baseline multiples; Helvetica Now/Inter/Suisse/Söhne/Neue Haas; **never justified**, never centered body. The discipline *is* the grid + restraint. Slop: Helvetica + a red square but *no actual grid*; everything centered.
- **Editorial / Magazine** — oversized display type, dramatic scale contrast, active whitespace, pull quotes, drop caps, serif+sans pairing, intentional broken grid. `font-size:clamp(2.5rem,8vw,7rem); line-height:.95; letter-spacing:-.02em` for display; body `max-width:65ch; line-height:1.6`. Reads expensive because of *hierarchy and rhythm*, not the typeface. Slop: Playfair Display + a centered headline + a stock hero photo (the default "elegant" AI tell).
- **Warm minimalism / neo-minimalism** — the headline 2026 movement and named successor to sterile flat/AI minimalism: keep the restraint and whitespace, add **warmth** — **earthy low-saturation palettes** (oat, cream, beige, milk-tea, terracotta, warm-gray, sage, ochre — OKLCH with low chroma ~.03–.07), **natural texture** (a whisper of SVG grain, paper/linen, soft noise), **character serifs** or humanist sans with real scale contrast, and gentle depth/soft shadows over hard flatness. Reads expensive *and* human (the deliberate anti-AI move) — basically the "expensive" recipe with a warm color story. Slop: beige-on-beige with no contrast or hierarchy (warmth as an excuse to skip the system), or one stock terracotta accent with nothing else changed.
- **Maximalism** — curated chaos: `mix-blend-mode` layering (Baseline widely available; can force stacking contexts + cost paint — isolate with `isolation:isolate`), 8–12 clashing saturated hues, eclectic type from 12px to 30vw, texture/marquees/stickers. Survives only with **recurring motifs + one anchor of order** (a grid or fixed nav). Examples: Spotify Wrapped, Liquid Death, Gucci editorial. Slop: random clutter with no repeating system.
- **Collage / scrapbook / zine** — layered analog *fragments*: torn-paper edges, tape strips, handwritten scribbles, sticker-style cutouts, photo cutouts and clashing textures with **visible seams** (overlaps, rough edges, intentional misalignment as the point). Distinct from neo-brutalism (raw structure) and maximalism (curated color chaos) — this is tactile, hand-assembled, nostalgic, pitched explicitly as the **antidote to soulless AI imagery** (Nike, Spotify, fashion microsites). Recipe: PNG cutouts with real torn/tape edges + slight rotations + `mix-blend-mode` + grain/paper texture; let one motif repeat so it reads composed, not messy. Keep real text in the DOM (don't bake copy into the collage) and mind contrast where fragments overlap. Slop: a few stock tape PNGs scattered on a plain grid with no compositional intent.
- **Y2K / Frutiger Aero** — Y2K = chrome/holographic/blobjects/pixel; Frutiger Aero = glossy aqua + bokeh + lush-nature photography + skeuomorphic gloss. Glossy button: high radius + `::before` top-half `white→transparent` shine + inner glow + grounded drop shadow; gradients behave like *light* hitting a surface. Holographic/iridescent now cheaply done with `linear-gradient(in oklch longer hue, …)` + `mix-blend-mode`. Slop: flat color + one fake gloss stripe and no light logic.
- **Vaporwave / Synthwave / Retrowave** — synthwave = sincere 80s nostalgia; vaporwave = ironic/glitch/pastel/Hellenistic-bust. **Layered neon glow** (white core → colored bloom via stacked `text-shadow`), perspective grid floor (CSS `transform: rotateX()` + `repeating-linear-gradient`), gradient sun with scanline cutouts. Slop: flat purple→pink + one grid PNG + Orbitron set huge.
- **Cyberpunk / Sci-fi HUD** — near-black + neon, mono type, **glitch** (chromatic aberration via offset red/cyan `::before`/`::after` copies, clipped with `clip-path` — `shape()`/`inset()` newly cross-browser early 2026), subtle scanlines (`repeating-linear-gradient` at ~.05–.07 alpha), **notched/angled `clip-path` HUD frames**, data-readout micro-type. Glitch is *occasional and triggered*; palette stays restrained (1–2 neons). Slop: neon-green Courier + a glitch animation looping forever.
- **Dark mode / Dark-tech** — charcoal not `#000` (`#0a0a0f`–`#141416`); convey elevation by **lighter surfaces / translucent white overlays** (`rgba(255,255,255,.03)`→`.06`→`.09`) not shadow; author colors in **OKLCH/LCH**; off-white text (~85–90% white, never pure), **desaturated** accent (chroma pulled down so it doesn't vibrate on dark); one sparse radial glow. Exemplars: Linear, Vercel, Raycast. Slop: pure `#000` + mid-grey text + a saturated purple gradient + a frosted card — the single most common vibe-coded "AI dark theme" tell.
- **Dithering / Halftone** — luminance-driven dot/pixel patterns. CSS (`radial-gradient` dot pattern + `mask`), SVG filters, canvas (sample image luminance per cell), or a **WebGL Bayer-matrix / blue-noise shader** for real-time/animated. Examples: Obys, Minh Pham, *Return of the Obra Dinn*. Monochrome or duotone; pairs with grain. A current "designy" signal when *generated*, dead when faked. Slop: a flat halftone PNG dropped on top as an overlay.
- **Grain / Noise** — the highest-ROI "expensive" tactic and the cheapest fix for gradient banding. Inline SVG `feTurbulence` (`type="fractalNoise"`, `baseFrequency` ~.6–.9, `numOctaves` 1–3) tiled at **opacity .03–.08** + `mix-blend-mode:overlay` (→ "grainy gradients"). Keep it static and subtle; an animated grain costs continuous repaint. Slop: visible/too strong, or a tiled PNG whose repeat seam shows.
- **Liquid / Fluid / Metaball** — SVG **goo filter** (`feGaussianBlur` + `feColorMatrix` alpha-contrast on a container) for 2D blob-merge; **WebGL** SDF metaballs (`smin` smooth-min) or refractive liquid glass (snapshot the page to a texture → refract UVs → add specular + chromatic aberration) for brand-level. CSS `backdrop-filter` **cannot refract** — it only blurs; true bending is WebGL/SVG-filter, CSS blur is the graceful fallback. Budget **one earned moment** + a `prefers-reduced-motion` static fallback.
- **Kinetic typography** — type *performs*: char/word reveals, horizontal scroll, weight/width morph on variable axes. Stack: **GSAP + ScrollTrigger + SplitText** (2025 rewrite: screen-reader-safe, `autoSplit`/`onSplit` re-split) **+ Lenis + variable fonts**. `clamp()` sizing; animate **`transform`/`opacity` only**; keep the underlying text selectable/readable; reduced-motion fallback. Slop: a generic per-character stagger on every heading in a default font.
- **Anti-design** — deliberate convention-breaking (system defaults, clashing type, broken grids, intentional "ugliness"). Works only with a clear point of view + strong content, and must stay **usable and accessible underneath** the rebellion. Neo-brutalism is its housetrained, commercial descendant.

**Pairings that work:** Swiss grid + editorial display + grain; dark-tech + mesh gradient (OKLCH) + grain + *restrained* glass; bento + flat + subtle depth; kinetic type + liquid + dark-tech; Y2K gloss + holographic OKLCH-hue gradients. **Clash (avoid):** neumorphism + anything needing real contrast/a11y; glass-on-everything + flat dark-tech (the slop signature); maximalism + bento (order vs chaos cancel out). **Dated tells:** neumorphism, literal-texture skeuomorphism (leather/wood), glass-on-everything, flat-purple-blob "mesh," constant-glitch cyberpunk. The named 2026 counter-movements to AI sameness are **warm/neo-minimalism** (warmth + texture over sterile flat) and **collage/scrapbook** (hand-assembled, tactile) — both above; reach for one when the brief is "make it feel human, not AI-made."

## §4. Scroll & Interaction Techniques

- **Parallax** — layers move at different rates. Order of preference: native `animation-timeline: view()`/`scroll()` (best — off main thread), GSAP `scrub`, then Lenis-driven `transform`. Avoid `background-attachment: fixed` (forces paint, broken/disabled on iOS Safari, janky). Subtle is the whole game: keep displacement small (≤10–15% of scroll distance); large parallax is a strong **vestibular** trigger → gate on `prefers-reduced-motion`.
- **Scroll-jacking** — overriding native scroll. Breaks velocity/momentum expectations, keyboard (PgUp/Dn/Space/Home/End), screen-reader virtual cursor, find-in-page, and scrollbar drag. Prefer **scrubbing** (animation is *driven by* the still-native scroll) over true hijacking; if you must hijack, give a visible progress affordance and a non-hijacked escape. Smooth-scroll libs (Lenis) are *not* scroll-jacking — they modify the real scroll position.
- **Scroll-triggered reveals** — **IntersectionObserver** (lightweight, fire-once with `obs.unobserve(el)`, toggle a class, let CSS do the transition; tune `rootMargin: "0px 0px -10% 0px"` and `threshold`) for one-shot enter animations; **GSAP ScrollTrigger** when tied to scroll *progress*, pinning, or a sequenced timeline (`ScrollTrigger.batch()` for many elements on one observer). Best of all where supported: native `animation-timeline: view()` (zero JS, zero main-thread cost). Never bind raw `scroll` listeners for reveals.
- **Scroll scrubbing** — animation progress bound 1:1 to scroll position. ScrollTrigger `{ scrub: true }` (locks to scrollbar) or `{ scrub: 0.5–1 }` (adds a smoothing lag in seconds — feels premium); or native `animation-timeline: scroll()`/`view()`. `scrub:<num>` + Lenis = buttery. Use `ease:"none"` on scrubbed tweens (the scroll *is* the easing).
- **Pinning** — **`position: sticky` first** (cheap, accessible, no layout hacks; but silently breaks if any ancestor has `overflow: hidden/auto/clip` or a `transform`/`filter`/`will-change` that creates a containing block). Reach for ScrollTrigger `pin: true` (clones into a **pin-spacer**, switches to `position: fixed`, preserves layout height) only when sticky can't express it (scrubbed reveals during the pin, horizontal tracks). Call `ScrollTrigger.refresh()` after fonts/images/async layout settle, and `invalidateOnRefresh: true` so end values recompute on resize.
- **Horizontal scroll** — pin a section, translate the track by viewport-relative distance:
  ```js
  gsap.to(track, {
    x: () => -(track.scrollWidth - innerWidth),
    ease: "none",
    scrollTrigger: { trigger: track, scrub: 1, pin: true, invalidateOnRefresh: true,
      end: () => "+=" + (track.scrollWidth - innerWidth) }
  });
  ```
  Always provide arrow/keyboard escape (a pinned horizontal section can trap keyboard users) and don't nest a real horizontal scroller inside (double-scroll confusion).
- **Smooth scroll — Lenis** is the standard (`lenis` **v1.3.x**, by darkroom.engineering; the old `@studio-freight/lenis` + `@studio-freight/react-lenis` packages are **deprecated** — use `lenis` and `lenis/react`). Wire it to the GSAP ticker (one rAF loop for both):
  ```js
  const lenis = new Lenis();              // default lerp:0.1
  lenis.on('scroll', ScrollTrigger.update);
  gsap.ticker.add((t) => lenis.raf(t * 1000));   // t is seconds → ms
  gsap.ticker.lagSmoothing(0);
  ```
  Config: `lerp` (~0.1, exponential ease toward target) **or** `duration` + `easing` (fixed-time) — not both. Use `autoRaf: false` when driving rAF yourself (the GSAP pattern above); `autoRaf: true` if you want Lenis to own the loop.
  - **React:** import from **`lenis/react`** — wrap in **`<ReactLenis root>`** and read the instance anywhere via **`useLenis(cb)`** (RAF tied to React's lifecycle, no prop-drilling). The package is now just **`lenis`** (Studio Freight → Darkroom Engineering; the old `@studio-freight/*` packages and the deprecated `smoothTouch` option are retired — installing `@studio-freight/lenis` lands you on the frozen 1.0.42).
  - **Lenis does not honor `prefers-reduced-motion` itself** → don't init it (or call `lenis.destroy()`) under reduced motion so the OS's native scrolling returns. Locomotive Scroll v5 is now built on Lenis.
- **Sticky stacking cards** — `position: sticky; top: 0` per card + scale/`border-radius`/opacity shift as later cards arrive. Native `animation-timeline: view()` can do the scale/fade-on-leave with zero JS. **New (Chrome 133+, no FF/Safari yet):** CSS **scroll-state container queries** (see below) — `container-type: scroll-state` + `@container scroll-state(stuck: top) { … }` styles a sticky card *while it's stuck* without any JS/IO; progressive enhancement, fall back to IO.
- **Scroll snapping** — CSS `scroll-snap-type: y proximity` (gentler — only snaps when you land near a point; **`mandatory` can trap users** mid-content and skip short sections) + `scroll-snap-align: start|center`; `scroll-padding-top` to clear a fixed header; `scroll-snap-stop: always` to prevent skipping past a slide. Prefer over JS snapping. `@container scroll-state(snapped: y)` (Chrome 133+) can style the currently-snapped child.
- **CSS scroll-state queries** (NET-NEW; Chrome/Edge **133+**, **144+** for `scrolled` — Chromium-only → progressive enhancement) — style descendants by a container's *scroll state*, no JS, off the main thread: put `container-type: scroll-state` on the scroller, then `@container scroll-state(stuck: top){…}` styles a `position:sticky` child **only while it's stuck** (shadow-on-stuck nav with zero scroll listeners), plus `scroll-state(snapped: x|y){…}` (style the snap child that's currently snapped) and `scroll-state(scrollable: top|…){…}` (content remains past that edge → auto-show scroll arrows). Directly replaces the classic shadow-on-stuck-header / snap-styling scroll-listener pattern. No Firefox/Safari → keep an IntersectionObserver/ScrollTrigger fallback.
- **Native scroll-driven animations** — `animation-timeline: scroll()` (scroll-progress) / `view()` (element's progress through viewport) + `animation-range` (`entry` / `exit` / `cover` / `contain`). Runs **off the main thread** → smooth under JS load, the cheapest scroll animation there is. **Support: Chromium 115+ and Safari 26+; Firefox has it implemented but behind a flag (`layout.css.scroll-driven-animations.enabled`, default-on in Nightly only) so NOT Baseline (~85%+ global).** Feature-detect `CSS.supports('animation-timeline: scroll()')` and fall back to IO/ScrollTrigger (Firefox workaround: `animation-duration: 1ms` so it renders the end-state). Always pair with a `@media (prefers-reduced-motion: reduce)` override.
- **CSS carousels** (NET-NEW; Chrome/Edge 135+, experimental → fallback required) — build accessible prev/next + dot-marker carousels from a `scroll-snap` container with **zero JS**. On the scroller: `::scroll-button(left){content:'‹'}` / `::scroll-button(right){content:'›'}` generate real focusable buttons (auto-disabled at the ends); `::scroll-marker-group` + `li::scroll-marker{content:''}` generate a focusable dot per item, with `:target-current` marking the active one. Because the generated buttons/markers are real focusable elements, it's an a11y win over hand-rolled markup. Chromium-only → feature-detect and fall back to a JS carousel.
- **Magnetic buttons / cursor** — on `mousemove`, translate the element by a fraction (~0.2–0.4) of the cursor's delta from center, ease back to 0 on `mouseleave`. GSAP `quickTo`/`quickSetter` for cheap per-frame writes (avoids creating tweens every event). Pointer-only (`@media (hover: hover) and (pointer: fine)`) → never gate function on it; disable under reduced motion. (Already on Motion? **Motion+ `Cursor`** ships this off the shelf.)
- **Custom cursors** — hide the native cursor, lerp a follower div (`transform: translate3d`). High cost to a11y if done carelessly: keep real `:focus-visible` states, full keyboard usability, and unchanged hit areas; respect `prefers-reduced-motion` (drop the lerp/lag) and consider showing the OS cursor on touch/keyboard input. Never hide the cursor over text inputs.
- **Hover distortion** — map the `<img>` onto a WebGL plane (OGL / gpu-curtains / Three) and drive shader uniforms (often a flowmap) with GSAP. Lazy-init on first hover/intersection; always keep a real `<img>` in the DOM as the fallback and for SEO/a11y. (curtains.js is legacy WebGL; **gpu-curtains** is the WebGPU successor.)
- **Marquee** — CSS: duplicate the content twice, `@keyframes { to { transform: translateX(-50%) } }`, `animation: scroll Ns linear infinite` (use `translate`, not `margin/left`). JS (GSAP `xPercent` + modulo wrap, or a Motion loop, or **Motion+ `Ticker`**) only when you need pause-on-hover, drag, or velocity coupling. Pause/stop under reduced motion (auto-motion >5s must be pausable — WCAG 2.2.2), `aria-hidden="true"` the duplicated copy, and respect `prefers-reduced-motion`.
- **Page transitions — View Transitions API** first. **SPA / same-document:** `document.startViewTransition(() => updateDOM())` + `view-transition-name` on shared elements for "magic move" — **Baseline Newly Available (Oct 2025)**, works cross-browser (Chrome/Edge 111+, **Firefox 133+**, Safari 18+). **MPA / cross-document:** `@view-transition { navigation: auto; }` — zero JS — is **Chromium + Safari only** (Chrome/Edge 126+, **Safari/iOS Safari 18.2+**); **Firefox does NOT ship it** (partial/disabled-by-default behind `dom.viewTransitions.enabled`, ~82% global), so treat Firefox as the no-transition fallback (it degrades gracefully — same-origin only). Style with `::view-transition-old(name)` / `::view-transition-new(name)`. **View-Transition *types*** (`startViewTransition({ types })` / `@view-transition { types }` + `:active-view-transition-type(name)`) let one transition pick a named animation (e.g. forward vs back) — cross-browser on the SPA surface since early 2026. Reach for Barba.js (~7KB) or Swup (both still actively maintained, mid-2026) for bespoke choreography or broader legacy support. Wrap transition triggers in a reduced-motion check.
- **FLIP** — First (measure start), Last (apply final state, measure), Invert (apply a `transform` that visually returns it to First), Play (transition the transform to identity). Only `transform`/`opacity` animate; layout is computed once → no per-frame reflow. Powers shared-element, list reorder, and expand/collapse. **GSAP Flip** (`Flip.getState` → mutate DOM → `Flip.from`) automates it for vanilla; **Motion**'s `layout`/`layoutId` is the React equivalent; the **View Transitions API is the browser-native FLIP**.

## §5. Micro-interactions & Motion Principles

- **Microinteractions (Dan Saffer):** Trigger → Rules → Feedback → Loops/Modes. Every meaningful state change earns a small, single-purpose response; the four parts keep it intentional rather than decorative.
- **Easing:** ease-**out** for entrances (fast start → gentle settle — feels responsive), ease-**in** for exits (lead the eye off), ease-in-**out** for on-screen moves, **linear** only for continuous motion (marquee/spinner/progress). Avoid the muddy CSS `ease` default for anything expressive. Tune custom beziers and store them as CSS vars: `--ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1)` (≈ easeOutExpo, the go-to "premium snap"), `--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1)`, `--ease-in-out: cubic-bezier(0.65, 0, 0.35, 1)`.
- **Spring physics:** `stiffness` (snappiness), `damping` (settle vs. bounce — under-damped overshoots, over-damped is sluggish), `mass` (weight/inertia). Springs feel natural because they carry velocity from gestures and are fully interruptible (no awkward jump on re-trigger). Use for direct-manipulation / interruptible UI (drag, toggles, layout); use duration+easing for deterministic, choreographed sequences where exact timing matters. Motion's springs are real solvers (define by `stiffness/damping/mass` or by `duration`+`bounce`).
- **Staggering:** 30–80ms per item, total cascade kept **< ~600ms** (longer reads as slow); `stagger: { from: 'center' | 'edges' | 'start' | 'random' }` for directionality. Drop or shorten stagger under reduced motion.
- **Orchestration:** one focal point per moment; lead the eye along a path; **overlap** (the next element starts before the previous finishes — ~70–80% overlap feels alive, sequential feels robotic); keep one consistent motion voice (shared easing + duration tokens) across the whole product.
- **12 Principles of Animation → UI:** squash & stretch (button press-squish), anticipation (a tiny wind-up before a big move), staging (dim/scrim the background for a modal), straight-ahead vs pose-to-pose, follow-through / overlapping action (stagger, slight overshoot-and-settle), **slow in / slow out (easing — the single most important one)**, arcs (curved rather than dead-straight paths), secondary action, timing (heavier objects move slower), exaggeration (sparingly), solid drawing (consistent light/depth), appeal (brand personality). The lesson: borrow the *intent*, not the cartoon amplitude.
- **Material motion:** Emphasized vs Standard easing sets and standardized durations (short/medium/long tokens). **M3 Expressive (announced May 2025, broad rollout through Android 16 / late 2025)** replaced duration-based easing with a **spring-based** motion system (spatial vs effects springs) and added shape-morph motion (35 new shapes, Material Shapes library). Container-transform / shared-element = FLIP / View Transitions under the hood.
- **Entry/exit without a JS lib (modern CSS):** `@starting-style` defines the "from" state on first render and `transition-behavior: allow-discrete` lets `display`/`overlay` participate, so popovers/dialogs/toasts can animate in *and* out with pure CSS — no `AnimatePresence`. **Baseline Newly Available** (Chrome/Edge 117+, Safari 17.5+, Firefox 129+); degrades to an instant show/hide where unsupported. Use for lightweight reveals; reach for Motion's `<AnimatePresence>` when you need spring exits, layout, or interruption.
- **Animate to `height:auto` — `interpolate-size` / `calc-size()`** (NET-NEW; Chrome/Edge 129+ only) — finally transition/animate **to `height:auto`** (and `fit-content`/`max-content`) with no JS measuring, for accordions/disclosures. Set once high in the cascade: `:root{ interpolate-size: allow-keywords }` then transition `height` from `0` to `auto`; use `calc-size(auto, size)` only when you also need math on the intrinsic size (it opts in implicitly). Chromium-only → degrades to an instant snap elsewhere (safe progressive enhancement). Pair with `@starting-style` + `transition-behavior: allow-discrete` to also animate `display:none`↔open.
- **States (the often-skipped layer):** skeleton/shimmer for perceived speed (render the *shape* of content, animate a gradient sweep; freeze to a static placeholder under reduced motion); **optimistic UI** via React 19's `useOptimistic` (stable; immediately render the expected result, auto-revert on failure) — **only** for high-success, cheap-to-roll-back actions (like, follow, reorder), never irreversible/financial ones; and deliberately design **empty / loading / error** states (a beautiful happy path with a default-grey error is unfinished).
- **Animated numbers — NumberFlow** (`@number-flow/react`; also Vue/Svelte/vanilla via `number-flow`): dependency-free, **accessible** rolling-digit/odometer transitions for counters, prices, stats, timers. Respects reduced motion by default (`respectMotionPreference` prop, `useCanAnimate()` hook), is `Intl.NumberFormat`-aware (currency/locale), and offers a `continuous` mode that passes through the in-between values. The correct default over a hand-rolled odometer (which is usually janky *and* inaccessible). In the Motion ecosystem, **Motion+ `AnimateNumber`** does the same with a `trend` prop for spin direction.
- **Haptics:** Vibration API (`navigator.vibrate([10])`) for short, meaningful confirmations. **Android Chrome/Edge/Samsung only — NOT iOS Safari (never implemented) and NOT Firefox (removed in 129+); ~77% global.** Requires a user gesture, mobile-only, must be brief; **never haptic-only** — pair with a visible cue. (iOS web has no public haptics API; don't rely on it.)
- **Don Norman:** affordance (what an element makes possible), signifier (the *perceivable* cue advertising it), feedback (a visible response within ≤100ms — Doherty threshold ~400ms for "feels instant" on completion), mapping (control ↔ effect correspondence). A control with no feedback feels broken even when it worked.
- **Timing defaults:** hover/press/toggle 120–200ms; reveal / small move 200–300ms; modal / page / large move 300–450ms; hard ceiling ~500ms (longer reads as sluggish); respond to *any* input ≤100ms. Exits are usually ~20–30% faster than entrances. "Motion should be felt, not seen."
- **Reduced motion (mandatory):** detect in CSS (`@media (prefers-reduced-motion: reduce)`) **and** JS (`matchMedia('(prefers-reduced-motion: reduce)')` + `.addEventListener('change', …)`). **Reduce, don't always nuke** — keep opacity fades and shortened durations; remove large translation, parallax, scroll-jack, marquees, auto-spin, and overshoot/bounce; replace motion-only signifiers with non-motion ones (color/border/icon). Prefer opting *into* motion (`@media (prefers-reduced-motion: no-preference)`). Maps to WCAG 2.3.3 *Animation from Interactions* (Level AAA) with the "essential motion exempt" carve-out; flashing content (>3×/sec) is forbidden regardless (2.3.1).

## §6. Typography

- **Modular scale:** base × ratio^n. Ratios: 1.2 (UI conservative — minor third), **1.25 (web UI workhorse — major third)**, 1.333 (perfect fourth, marketing), 1.414 (augmented fourth/√2), 1.5 (perfect fifth, bold), 1.618 (golden/dramatic, editorial display). Use [type-scale.com](https://type-scale.com). **Fewer sizes (5–7) = more consistent.** Pro move: a *single* ratio over a wide range explodes (1.618^7 ≈ 19×) — either cap the steps or use a **different (smaller) ratio per breakpoint** so mobile doesn't blow out (the Utopia approach below).
- **Variable fonts:** axes `wght`/`wdth`/`slnt`/`ital`/`opsz`/`grad` (registered, lowercase) + custom (UPPERCASE, e.g. `GRAD`, `MONO`). One file delivers the whole 100–900 range with smooth interpolation — drops payload from ~320KB (four static weights) to ~110KB (one variable file), worth ~0.3–0.5s LCP on 4G. High-level props (`font-weight:550`, `font-stretch:75%`, `font-style:oblique 10deg`, `font-optical-sizing:auto`) cascade and inherit normally; **`font-variation-settings` is all-or-nothing — it does NOT inherit per-axis** (setting one axis resets the others to default) → store each axis in its own CSS var and recompose the whole string. Don't treat axis tweens as cheap GPU work: axis changes trigger relayout/repaint, so animate sparingly (one heading, not a wall of text). Match `opsz` to the rendered size for correct optical sizing; `font-optical-sizing:auto` does this for you.
  ```css
  :root { --wght:400; --opsz:14; --GRAD:0; }
  h1 { --wght:680; --opsz:96; font-variation-settings:"wght" var(--wght),"opsz" var(--opsz),"GRAD" var(--GRAD); }
  ```
- **Kerning/tracking/leading:** `letter-spacing` in `em` (scales with size); `line-height` **unitless** (so it scales per-element). Tracking inversely scales with size (display `-.02em` to `-.04em`; small-caps + all-caps `+.05em` to `+.1em`; body ~0). Leading: body 1.5–1.7, display 0.9–1.1. Turn pair-kerning on with `font-kerning:normal` (default `auto` already enables it at most sizes).
- **Cap-height trimming (new):** `text-box-trim:trim-both; text-box-edge:cap alphabetic` removes the font's built-in over/under leading so a heading's *cap height* (not its line box) aligns to your spacing — kills the "why is there mystery space above my H1" problem and makes optical centering trivial. Shorthand `text-box: trim-both cap alphabetic`. **Chrome/Edge 133 (Feb 2025) + Safari 18.2 (Dec 2024); NOT Firefox (none through 154) → ~79% global, still Working Draft, not Baseline.** Progressive enhancement; layouts must not depend on it. (Until then, the old `line-height` + negative-margin hack.)
- **Fluid type:** `clamp(min, intercept_rem + slope*100vw, max)`. slope=(max−min)/(maxVw−minVw); intercept=min−slope×minVw (in rem). **Keep a `rem` term** so the page still scales when the user zooms — pure `vw` fails WCAG 1.4.4 (text must reflow/scale to 200%). Use [utopia.fyi](https://utopia.fyi) (lets the ratio differ at each breakpoint and emits ready `--step-n` custom props, e.g. `--step-0: clamp(1.125rem, 1.07rem + 0.23vw, 1.25rem)`). Tailwind v4: do it in `@theme` or with the `tailwind-utopia` plugin.
- **Line-wrapping (high-ROI):** `text-wrap: balance | pretty` — `balance` evens out the last few lines of **headings / short blocks** (≤~6 lines in Chromium, then it bails for perf) so there's no lonely one-word last line; `pretty` improves **body** by preventing orphans/runts and improving the rag. `balance` is **Baseline 2024** (Chrome/Edge 114+, FF 121+, Safari 17.5+) — the zero-JS replacement for headline-balancer libs; `pretty` is Chrome/Edge 117+, **Safari 26+**, **still NOT Firefox** (~82% global, mid-2026) → pure progressive enhancement (`@supports`). Both fail gracefully to normal wrapping. Apply `balance` to `h1–h3`/`.lead`, `pretty` to `p`.
- **Measure (line length):** body **60–75ch** (`max-width:65ch` is the safe default); too long tires the eye, too short breaks rhythm. Set on the text element, not the container.
- **Pairing:** contrast of *role* + harmony of *mood*; **superfamily pairing is safest** (one family's sans + serif + mono share proportions/metrics — IBM Plex, Source, Recursive, Fraunces+its sans). Common structure: one display + one body + one mono. Limit to **2–3 families** total. Tools: Fontjoy, Fontpair, [Typewolf](https://www.typewolf.com) (see pairings in the wild). Default-avoidance: if the brief screams "default," it's Inter+Playfair or Roboto — pick something with a point of view instead.
- **OpenType / figures:** prefer the cascading `font-variant-*` longhands over the low-level `font-feature-settings`: `font-variant-numeric:tabular-nums slashed-zero` (tables/dashboards — equal-width digits that don't jiggle as values change), `oldstyle-nums` (running prose), `font-variant-caps:all-small-caps` (REAL small caps with correct weight, **not** `text-transform:uppercase` which just shrinks caps and looks anemic), `font-variant-ligatures`. Stylistic sets still need `font-feature-settings:"ss01" 1` (and `"cv01"` character variants). `dlig` (discretionary ligatures) display-only. Gotcha: `font-feature-settings` is also all-or-nothing — listing one feature disables the defaults, so prefer `font-variant-*` whenever the feature has a longhand.
- **Foundries — free quality:** **[Fontshare](https://www.fontshare.com)** (Indian Type Foundry; Satoshi, General Sans, Clash Display, Switzer, Cabinet Grotesk — ITF Free Font License, commercial use included, variable files), **Google Fonts** (Inter, **Geist**/**Geist Mono** + the new **Geist Pixel** bitmap face — announced Feb 6 2026, OFL 1.1, 5 grid variants (Square/Grid/Circle/Triangle/Line) built on the Geist Sans/Mono foundations, on npm `geist` + Google Fonts; Plus Jakarta Sans, Fraunces, Newsreader), **Pangram Pangram** (generous free tier). Premium — **Klim** (Söhne — Akzidenz-Grotesk reborn, by Kris Sowersby; Tiempos, Calibre), **Grilli Type** (GT America — 84 styles bridging US Gothics + Euro neo-grotesks; GT Sectra, GT Flexa), **ABC Dinamo** (ABC Diatype, ABC Monument Grotesk, ABC Marfa). "Expensive" signals = large optical ranges, real small caps + multiple figure styles, restrained-but-distinctive grotesques (vs. anything you've seen on 1,000 sites).
- **Loading & layout:** self-host or preload the variable file, `font-display:optional` (no FOUT swap mid-session — text either shows in the custom font fast or stays in the metric-matched fallback for that load; best CLS) or `swap` + a fallback tuned with `size-adjust`/`ascent-override` so the swap doesn't shift layout. `subset` to the glyphs you use.
- **"Expensive" recipe:** 1–2 families; body line-height 1.5–1.7, display 1.0–1.1; tight display tracking (`-.02em`→`-.04em`); ratio ≥1.25 with real scale contrast (don't make H1 only 1.2× the body); measure 60–75ch; body 16–20px; OpenType figures on; `text-wrap:balance` on heads + `pretty` on body; and above all **restraint** — the system, executed, not five fonts and ten sizes.

## §7. Color

- **Schemes:** complementary (max contrast/tension), analogous (calm, harmonious), triadic (vibrant/balanced), split-complementary (safer contrast than full complementary), monochromatic (one hue, vary L/C), duotone (two-color mapping). Most "expensive" UIs are effectively **monochrome + one accent** (see Recipes).
- **Spaces:** **HSL lies about lightness** — equal `L%` across hues are *not* equally bright (yellow at 50% reads far lighter than blue at 50%), so HSL ramps come out uneven and muddy. **OKLCH** (cylindrical form of Oklab) is **perceptually uniform** (equal L steps look equally spaced), **P3-capable** (vivid colors sRGB can't show), and **hue-stable** (changing L/C doesn't shift the hue — no purple drifting blue). Syntax `oklch(L C H / a)` — **L 0–1** (or 0–100%), **C 0→~0.37** in practice (unbounded in spec; `100%` maps to 0.4, in-gamut sRGB tops out near 0.37, P3 a bit higher), **H 0–360**. **Relative color** derives variants without a picker: `oklch(from var(--c) calc(l - .08) c h)` (one step darker, same hue/chroma). Support: base **`oklch()` ~93%** (Chrome/Edge 111, Safari 15.4, FF 113; widely available since 2023); **relative-color syntax ~86%** (84.23% + 2.39% = 86.62%; Chrome/Edge 122, Safari 18.0, FF 128) — both production-safe with a fallback declaration before them.
- **`color-mix()`** (Baseline **Widely Available Nov 9 2025**) — blend two colors in a chosen space at runtime: `color-mix(in oklch, var(--brand) 85%, black)` for a hover shade, `color-mix(in oklch, var(--brand) 12%, transparent)` for a token-derived tint/scrim. Mix **in `oklch`/`srgb-linear`**, not the default sRGB, to avoid muddy midpoints. This is how you generate state/tint ramps from semantic tokens without Sass — and the pure-CSS way to tint gray text toward a colored background (the Refactoring UI rule). Tailwind v4 uses it under the hood for opacity modifiers.
- **`light-dark()`** (Baseline **Newly available** since May 2024 — all engines; Widely-Available promotion projected ~late 2026) — set both themes inline without a media query: `color: light-dark(oklch(.2 0 0), oklch(.93 0 0))`. **Requires `color-scheme: light dark`** on `:root` (and respects a manual `color-scheme` override), so it follows a user/UA theme toggle for free. The cleanest way to express semantic dark-mode tokens; keep dark accents desaturated ~10–15% regardless.
- **`@property` (typed custom properties)** (Baseline **Newly available Jul 9 2024** — all engines, Firefox 128 the last to ship) — registering a type unlocks **animation/transition of values plain `--vars` can't interpolate** (they parse as strings): `@property --angle { syntax:"<angle>"; inherits:false; initial-value:0deg }` then `@keyframes spin { to { --angle:360deg } }` animates a `conic-gradient(from var(--angle) ...)`; the same trick animates gradient **color stops** (`syntax:"<color>"`) and `<number>`/`<percentage>` driving effects. The native, compositor-friendly way to do animated gradient heroes / conic loaders without JS (and it's *why* Tailwind v4 requires `@property` + OKLCH-capable browsers). Also gives custom props a real `initial-value` + inheritance control.
- **Tokens / ramps:** two tiers — **primitive** (`blue-50…950`, raw values) + **semantic** (`--bg`/`--surface`/`--text`/`--text-muted`/`--primary`/`--border`/`--danger` referencing primitives; theming and dark mode remap the *semantic* layer, never the components). Build perceptually-even ramps directly in OKLCH: **fix H**, sweep **L evenly** (≈0.97 → 0.20 across 50→950), and **curve C** (low at the light/dark ends, peak in the mid ~500/600) so the ramp stays vivid without the ends going neon. Generators: Tailwind v4 ships OKLCH ramps; uicolors.app, Leonardo (contrast-target driven). **Radix 12-step roles** (memorize — it maps tokens to jobs): 1 app bg, 2 subtle bg, 3 component bg (normal), 4 component bg (hover), 5 component bg (active/selected), 6 subtle border/separator, 7 element border + focus ring, 8 strong/hovered element border, 9 **solid** (highest-chroma step, purest — primary buttons/fills), 10 solid hover, 11 low-contrast text, 12 high-contrast text. Steps 11/12 are *guaranteed* to APCA **Lc 60 / Lc 90** against a step-2 bg from the same scale (i.e. they pass text contrast by construction).
- **Contrast — ship to WCAG 2.2 AA:** **4.5:1** body text, **3:1** large text (≥24px, or ≥18.66px bold) and UI components/graphics/focus indicators (WebAIM checker; or in-browser DevTools). **APCA** (perceptual, *polarity-aware*, size/weight-aware; Lc 90 fluent body / 75 min body / 60 large / 45 headlines / 30 non-text floor) models dark-mode contrast far better than 2.x ratios — **but it is not a conformance target.** APCA was **removed from the WCAG 3 working draft in mid-2023** (reverted to "placeholder," exploratory items dropped) and WCAG 3's contrast method is currently "to be determined"; the W3C says WCAG 3 **won't reach Recommendation before 2028** (realistically later — multi-year). Worldwide law/procurement still cites WCAG 2.x AA, and using APCA *in place of* it creates legal risk. So: *design/eyeball* dark themes with APCA, but **certify with WCAG 2.2 AA**.
- **`contrast-color()` (NEW — Baseline Newly Available Apr 2026):** let the browser pick the readable foreground. `color: contrast-color(var(--brand))` returns **black or white** (whichever has more contrast against the input; ties → white) — kills the "is white text readable on this dynamic accent?" problem for buttons/badges/chips with user- or data-driven backgrounds. **Chrome 147 / Firefox 146 / Safari 26, all passing WPT.** Limits: only black/white for now (no arbitrary palette, no chosen target ratio yet) — for richer auto-theming, generate tokens at build time. Provide a static fallback color first for older engines (`@supports`).
- **Gradients:** **interpolate in OKLCH** to avoid the grey/muddy midpoint sRGB produces: `linear-gradient(in oklch, var(--a), var(--b))`. For polar spaces pick a **hue path**: `shorter hue` (default, nearest way around the wheel), `longer hue` (the long way — for rainbow sweeps), `increasing`/`decreasing hue` (force clockwise/counter-clockwise when *direction* matters more than distance), e.g. `linear-gradient(in oklch longer hue, red, red)` for a full spectrum loop. Always **add grain/dithering** to kill 8-bit banding on large soft gradients (see §3 Grain). For *animated* gradients use `@property` (above) rather than JS.
- **Tools:** Coolors (fast ideation), Adobe Color (wheel + built-in contrast + colorblind sim), **Realtime Colors** (preview a palette on a real landing layout — the single best "will this actually work" check), Huemint (AI, context-aware palettes), Tailwind (OKLCH default ramps), **Radix Colors** (12-step, light/dark/P3 wide-gamut, contrast-guaranteed), Open Color, Leonardo (generate ramps to hit a target contrast), uicolors.app (hex → 50–950).
- **Recipes:** **muted = low chroma** (C ≈ .03–.08 — the difference between "refined" and "crayon"); monochrome + **one** saturated accent (the accent = the CTA, used once per view); **60-30-10** (dominant / secondary / accent split); **dark mode needs accents desaturated ~10–15%** and slightly lighter — full-saturation hues *vibrate* and bloom against dark surfaces. In dark mode, build elevation from **lighter surfaces / translucent-white overlays** (≈.03→.06→.09 alpha), not drop shadows, and never use pure `#000` (charcoal `#0a0a0f`–`#141416`). Never convey state by **color alone** — pair with an icon/label/shape (WCAG 1.4.1 + colorblind users).

## §8. Layout & Composition

- **Mental model:** **Flexbox = 1D, content-out** (navs, toolbars, button rows, tag/chip wrapping, card internals — sizes from content, `flex-wrap` for reflow); **Grid = 2D, layout-in** (page templates, dashboards, bento — you define the structure, content fills it). Don't force one to do the other's job (a faux-grid built from flex `%` widths + margins is the classic tell). **Subgrid** (`grid-template-columns: subgrid`) inherits the *parent's* track lines so children align across siblings — **Baseline Widely Available 2026-03-15** (usable since Sep 2023, now ~95%+ global; Chrome/Edge 117+, Safari 16+, Firefox 71+). Killer use: a row of cards where every header/body/footer lines up on shared rows even though content lengths differ — set the card to `grid-row: span 3` and `grid-template-rows: subgrid`.
- **Grid setup:** 12-col `repeat(12, 1fr)` + `gap` + `span`; **named lines** (`[full-start] … [content-start]`) or `grid-template-areas` for app shells. Area rules: every named area must form a **rectangle**, `.` = empty cell, redefine the whole template per breakpoint. Prefer `gap` over margins (collapses cleanly, no edge double-spacing). Implicit tracks via `grid-auto-rows` / `grid-auto-flow: dense` (dense backfills holes — but it reorders *visually only*, so it can desync from DOM/tab order; see the order gotcha below).
- **Intrinsic sizing:** `min-content` / `max-content` / `fit-content(x)` / `minmax(min, max)` / `clamp()` inside tracks. The no-media-query responsive grid: `grid-template-columns: repeat(auto-fit, minmax(min(250px, 100%), 1fr))` — the inner `min(…, 100%)` prevents overflow on viewports narrower than the floor. **`auto-fit` collapses empty tracks so existing items stretch to fill the row; `auto-fill` keeps the empty tracks reserved** (items stay their min width, column rhythm preserved). Use `auto-fit` for "fill the row," `auto-fill` for "keep the column grid."
- **Masonry (emerging, NOT Baseline):** the staggered-tile / waterfall layout. The CSS WG settled the long "is it grid or its own thing" debate on **`display: grid-lanes`** (reuses grid templating + placement). **Safari shipped it first — `display: grid-lanes` is in Safari 26.4** (and exposed in Web Inspector with placement-order overlays); **Chromium prototyped `display: masonry` behind a flag in 140 and is migrating to `grid-lanes`**; Firefox had a `display: grid` masonry prototype and is switching too. Packing is tuned with **`flow-tolerance`** (default ~`1em`; accepts `lh` etc.) — it loosens/tightens the "place each item in the shortest lane" rule so visual order doesn't zig-zag away from tab order (an a11y win). (`flow-tolerance` is the name WebKit's shipped implementation uses; an earlier `item-flow` shorthand proposal — `<item-direction> || <item-wrap> || <item-pack> || <item-slack>` — is still churning and is NOT what ships.) Until it's Baseline, do masonry with a JS lib or CSS `columns` and **feature-detect** (`@supports (display: grid-lanes)`); unsupported browsers fall back to normal grid auto-placement. Native grid-lanes preserves **DOM order** (good for a11y), unlike most JS masonry libs.

```css
/* portable card grid: components size to themselves, no breakpoints */
.grid { display: grid; gap: 1.5rem;
  grid-template-columns: repeat(auto-fit, minmax(min(18rem,100%), 1fr));
  grid-auto-rows: auto auto 1fr auto; } /* parent MUST define the shared row tracks or rows-subgrid has nothing to inherit */
.card { display: grid; grid-row: span 4; grid-template-rows: subgrid; } /* header/body/footer align across the row */
```

- **Container queries** (size: Baseline 2023, now **Widely Available**, ~95%+): `container-type: inline-size` (+ optional `container-name`) on the parent, then `@container sidebar (min-width: 400px) {…}` → the component responds to **its own container, not the viewport** = genuinely portable components (one card that adapts in a sidebar, a grid cell, or full-bleed). Units: **`cqw/cqh/cqi/cqb/cqmin/cqmax`** — prefer **`cqi` (inline) / `cqb` (block)** over `cqw/cqh` (writing-mode-safe). **Style queries** (`@container style(--variant: hero)`) are now cross-browser — **Baseline Newly available** after **Firefox 151** completed custom-property `style()` queries (Chrome/Edge 111+, Safari 18+ already had them); use them to fork on a custom-property value, not just size. Gotcha: a container can't size to its own contents on the queried axis (it'd loop) — query `inline-size`, let block size grow.
- **Anchor positioning** (Baseline Newly available early 2026 — Chrome 125+, Safari 26+, **Firefox 147+ by default, Jan 13 2026**) — tether an absolutely/fixed-positioned element to a reference element in pure CSS, no JS measuring. `anchor-name: --menu` on the anchor; on the positioned element `position-anchor: --menu` + `top: anchor(bottom)` / `left: anchor(right)`, or the shorthand `position-area: bottom span-right`; `@position-try` + `position-try-fallbacks` auto-flips when it would overflow (and `anchor-center` centers on the anchor). The native replacement for Floating UI / Popper, and the canonical partner of the **Popover API** (§9) for tooltips/menus/comboboxes without a JS lib. Caveats: `position-area` overflow handling still differs Chrome vs Safari (test both); `@position-try` needs Safari 18.4+; OddBird ships a polyfill for shadow-DOM cases.
- **Viewport units** (Baseline **Widely Available June 2025**, ~95%+): **`svh`** = smallest (mobile bars expanded) → use for hero `min-height` so content never clips on first paint; **`lvh`** = largest (bars collapsed); **`dvh`** = dynamic, updates live as the URL bar hides — correct for fixed overlays/bottom sheets, but it reflows mid-scroll so avoid on the page hero (jank). `vi`/`vb` are the logical (writing-mode-aware) equivalents of `vw`/`vh`. iOS Safari note: `dvh` resizing fires after the gesture, not during — don't pin precise animations to it.
- **`field-sizing: content`** — let a `<textarea>` / `<input>` / `<select>` **auto-grow (shrink-wrap) to its content** (cap with `max-block-size` / `min-block-size`), killing the JS auto-resize-textarea pattern and its layout jank. **Chrome/Edge 123+, Safari 26.2+ (late 2025), and Firefox 152 (~June 2026) have all shipped it** → now in all three engines but **not yet flagged Baseline as of mid-2026** (~79% global; MDN/`web-features` lag the last engine landing). Progressive enhancement: older engines keep fixed sizing, so it degrades cleanly.
- **Spacing:** **8-point grid** (multiples of 8 — most screen widths/DPRs divide cleanly, fewer half-pixels) with a **4pt sub-step** for tight pairs (icon↔label, input padding). Expose as named scale tokens (`--space-1…`), never magic numbers. Reserve media proportions with **`aspect-ratio` (16/9, 1/1)** so images/embeds don't shift layout → directly protects **CLS ≤ 0.1**.
- **Whitespace** — the #1 "expensive" signal, and the cheapest. Macro (between sections → structure, calm, confidence) + micro (line-height, padding → legibility). It's *active* space (shaping relationships), not leftover. Refactoring UI rule: **start with too much, then remove.** Cramped + uniform spacing is the templated-AI tell.
- **Hierarchy:** rank with **size, weight, color/contrast, position, and whitespace** → one dominant focal point per view; **one accent = the primary CTA** (everything else secondary / ghost / text). De-emphasize by *lowering contrast* (lighter gray, tinted-on-color), not only by shrinking.
- **Composition:** rule of thirds (place focal points off-center on the 1/3 lines), golden ratio (~1.618 for pleasing proportions / sidebar splits), **F-pattern** (text-dense / scannable pages — weight the top bar + left edge), **Z-pattern** (sparse landing / hero — logo top-left → nav/CTA top-right → diagonal → primary CTA bottom-right).
- **Gestalt → UI:** proximity (group by spacing — the cheapest grouping tool, beats borders), similarity (one consistent style per control type), closure (skeletons, a peeking next-carousel card implying more), continuity (left-aligned fields create a clean scan line), figure/ground (modal + scrim/backdrop), **common region** (a card/box/section — a *stronger* group cue than proximity alone), focal point (the single accent CTA wins attention).
- **Asymmetry / broken grid** — intentional imbalance counterweighted by visual mass (a big image vs. a dense text block); reads as editorial/energetic, never random. Anchor it to a real grid you're deliberately breaking, not the absence of one.
- **Order vs. tab/reading order (a11y gotcha):** flex `order`, `row-reverse`, `grid-auto-flow: dense`, and explicit grid placement change *visual* order but **not** DOM/keyboard/screen-reader order → can produce a focus path that jumps around (WCAG 2.4.3 Focus Order, 1.3.2 Meaningful Sequence). Emerging fix: **`reading-flow` / `reading-order`** (Chrome 137+, May 2025; **Chromium-only, NOT Baseline**) realigns sequential focus to the visual layout of a flex/grid/block container. Until it ships broadly, **keep DOM order == intended reading order** and reorder with layout only when they still match.
- **Page structure:** above-the-fold must communicate value + expose the primary CTA without scrolling; hero sized `min-height: 100svh` (often *less* — full-bleed isn't mandatory and wastes the fold); reserve all media with `aspect-ratio`; lead with content, layer flourish on top.

## §9. Design Systems & Component Libraries

- **shadcn/ui** — copy-paste **source into your codebase** (not an npm dep) via CLI + registry; headless primitive (Radix *or* Base UI) + Tailwind v4 + CSS-variable theme tokens; `cva` variants, `cn()` (`clsx` + `tailwind-merge`). You own the code → unlimited customization, no version lock, no bloat. The current greenfield default. Tradeoff: you maintain copied code (re-pull for fixes); it's a starting point, not a finished system.
  - **CLI v4 (Mar 2026)** is the big jump: `shadcn init` scaffolds **full projects** for Next.js, Vite, React Router, Astro, TanStack Start, Laravel (+ `--monorepo`; dark mode wired for Next.js/Vite); `--base radix|base` picks the primitive layer; **`shadcn docs`/`shadcn info`** feed component code/API/install context to coding agents (`info` now reports framework, version, CSS-vars status, installed components); `--dry-run`/`--diff`/`--view` preview registry additions before write. **Presets** pack a whole design system (colors, theme, fonts, radius, icon set) into a shareable code (`init --preset` reconfigures an existing app, components included). **`shadcn/skills`** + the **MCP server** (since CLI 3.0, Aug 2025) make it agent-native; `registry:base` distributes complete systems in one payload; fonts are now a first-class registry type. Built-in style variants are named (e.g. **Luma** default / **Rhea** compact-density). **GitHub registries** (Jun 2026): install straight from any public repo (`shadcn add <user>/<repo>/<item>`) with no `shadcn build` step. ([CLI v4 changelog](https://ui.shadcn.com/docs/changelog/2026-03-cli-v4))
  - **Registries are now namespaced + MCP-driven** — pull items from many sources via one CLI with `@namespace/name` (`@shadcn/button`, `@v0/dashboard`, `@ai-elements/input`); the registry index is decentralized, so you can publish/consume any namespace (components, hooks, themes, even AI prompts). An official **shadcn MCP server** lets agents browse and install registry items directly. Reinforces shadcn as the greenfield default and the *distribution platform*, not just a component set.
- **Radix UI** — Primitives (unstyled, accessible, the WAI-ARIA reference) vs Themes (pre-styled). Now ships as **one unified `radix-ui` package** (latest **1.5.x**, mid-2026) instead of dozens of `@radix-ui/react-*` deps — shadcn's `new-york` style imports from it. Still huge adoption, but **dev slowed post-WorkOS acquisition** (complex bits like Combobox/multi-select lag) — the gap that let Base UI rise.
- **Base UI** — unstyled accessible primitives by the **MUI team** (also ex-Radix/Floating UI maintainers), Radix-like API, **~35 components**. **v1.0 stable shipped Dec 11 2025** (package renamed `@base-ui-components/react` → **`@base-ui/react`**); **v1.1 (Jan 15 2026)** improved Autocomplete/Combobox/Select and it's **actively expanding**; now at **v1.5.0 (mid-May 2026)**, developed full-time (v1.4.0 Apr 13 2026 was a bug-fix release). Smaller bundle, statechart-free, modern styling hooks → strong new default headless layer while Radix dev stays slow post-WorkOS; shadcn supports it via `--base base`. [Re-confirm exact latest patch at ship time.]
- **Tailwind v4** — utility-first, zero-runtime. CSS-first config (`@import "tailwindcss"; @theme {}`, no `tailwind.config.js` by default), **Oxide** (Rust) engine (~3.5–5× full / **100×+** incremental; trivial rebuilds in microseconds vs ~35ms in v3.4), native CSS vars for every token, OKLCH palette, container queries + `@starting-style`/`@property` built in. **v4.0 = Jan 2025; latest v4.3 (May 2026)** added scrollbar utilities, `@container-size` (height-aware containers), `zoom-*`/`tab-*`, four new neutral palettes (mauve/olive/mist/taupe), more logical-property utils, `font-features-*`, and a first-class webpack plugin. v3→v4 is breaking; `npx @tailwindcss/upgrade` codemods most of it. ([v4.3 blog](https://tailwindcss.com/blog/tailwindcss-v4-3))
- **Native platform primitives now do real component work (prefer over hand-rolled state):**
  - **Popover API + Invoker Commands** (NET-NEW; Invokers Baseline ~Dec 2025–Jan 2026: Chrome 135, Firefox 144, Safari 26.2) — native top-layer overlays with light-dismiss + focus handling, **zero-JS** wiring. Markup: `<button popovertarget="m">Open</button> <div id="m" popover>…</div>` (`popover="auto"` light-dismisses; `manual` doesn't). **Invoker Commands** drive native `<dialog>`/popovers declaratively: `<button command="show-modal" commandfor="dlg">` (also `close`, `request-close`, `toggle-popover`/`show-popover`/`hide-popover`; custom `--*` commands fire a `CommandEvent`). Browser handles top-layer, focus, a11y, Esc — pair with **anchor positioning** (§8) for a fully-CSS menu/tooltip and prefer it over a hand-rolled overlay-state hook.
  - **`field-sizing: content`** (NET-NEW) — form controls shrink-wrap and **auto-grow with their content** (the no-JS auto-growing `<textarea>`, content-width `<input>`/`<select>`); pair with `min-`/`max-` constraints, Tailwind ships `field-sizing-content`. As of mid-Jun 2026 it reaches **all three engines and is on the cusp of Baseline**: Chrome/Edge 123+ (Mar 2024), **Safari 26.2 stable** (desktop + iOS, Dec 12 2025), **Firefox 152 stable** (Jun 16 2026; Beta/Nightly before that). MDN still labels it *not Baseline* (it was Firefox-blocked until 152). It finally retires the classic JS auto-resize hack — but keep a fallback where pre-152 Firefox / older Safari still matter.
- **Headless UI** (Tailwind team) — small set of accessible interactive primitives (Menu, Combobox, Dialog, Tabs, Disclosure) for React/Vue. Narrow scope; reach for Radix/Base UI when you need more components.
- **HTML-first / framework-agnostic shadcn** (the shadcn look without React):
  - **Basecoat UI** (`hunvreus/basecoat`, MIT, ~v0.3.11) — the shadcn aesthetic on **Tailwind CSS + a sliver of vanilla JS** for the interactive bits (dialog/dropdown/toast), **no React, no Alpine**. Semantic class API (`btn`, `card`, `input`) instead of long utility strings; dark mode + a11y built in; theme-compatible with shadcn's CSS variables (drop in a tweakcn theme). The answer when a server-rendered app (Rails/Laravel/Django/Flask/Astro/plain HTML) wants the shadcn look without a framework.
  - **Franken UI** (`franken-ui/ui`, MIT, **v2.1** — Tailwind v4 support; v2.0 Feb 2025) — **HTML-first** shadcn-inspired kit on **UIkit 3 + LitElement web components**, so it runs in **any framework or none** (Vite plugin compiles the CSS). Use it (vs Basecoat) when you want UIkit's larger interactive component set as custom elements rather than CSS classes. Caveat: the maintainer is now building **Frankenstyle** as its "spiritual successor" — fine to adopt today, but watch the migration path before betting long-term.
- **Higher-order / complex components on top of shadcn:**
  - **Kibo UI** (`haydenbleasel/kibo`, MIT, ~v1.1.5) — a shadcn **registry of higher-order, composable components** that fill the gaps shadcn's primitives deliberately leave: **Gantt, Kanban, code block, color picker, dropzone, AI-chat input, rich editor**, plus blocks. Uses shadcn's CSS variables (themes/dark mode just work), wraps messy logic from various headless libs, and installs **own-the-code** via the shadcn CLI (namespaced) or its own — with an **MCP server** for agents. Reach for it when you'd otherwise hand-roll a complex widget on top of shadcn; complements (doesn't replace) the primitive layer.
- **Animated kits** (pair with shadcn, marketing pages): **Aceternity UI** (~100+ components now, most "wow," most overused), **Magic UI** (150+ landing blocks), **Motion Primitives**, **Cult UI** (AI-app patterns), **HextaUI**. - **Batteries-included** (enterprise/data-heavy, fast teams, no designer):
  - **MUI (Material UI)** — **v9 is current** (v9.0 Apr 8 2026; v8 existed, bumped to align with **MUI X v9** which versions in lockstep). ## §13. Performance & Accessibility

**Rendering pipeline:** Style → Layout → Paint → Composite. The property you animate decides how far back the browser rewinds each frame — pick one that stays late in the pipeline:

| Type | Examples | Cost |
|---|---|---|
| Composite-only | `transform`, `opacity`, `filter`*, `clip-path`* | **Cheap** (GPU reuses the layer's texture; no main-thread work) |
| Layout (reflow) | `width/height/top/left/margin/padding/font-size` | **Expensive** (recompute geometry → repaint → recomposite) |
| Paint | `color/background/box-shadow/border-radius` | Moderate (re-raster the layer, no relayout) |

*`filter`/`clip-path` are GPU-accelerated but still trigger a paint on change and can be costly at large sizes — cheap to *move/fade*, not free to *animate the blur radius*.

- **Frame budget:** 16.67ms @60Hz, 8.33ms @120Hz; you own ~10–12ms of the 60Hz frame (browser keeps the rest). **Consistency beats peak FPS** — a steady 60 reads smoother than 120-that-stutters. 120Hz/ProMotion is now common on phones/laptops; don't assume 60.
- **Layout thrashing (forced synchronous reflow):** reading geometry (`offsetWidth/Height`, `getBoundingClientRect()`, `getComputedStyle()`, `scrollTop`) *after* a write in the same frame forces the browser to flush layout early. **Batch all reads, then all writes**; schedule visual writes in `requestAnimationFrame`. Libraries like `fastdom` formalize this; React's `useLayoutEffect` reads before paint.
- **Layers:** `will-change: transform` (or legacy `transform: translateZ(0)`) promotes to a GPU layer — but each layer costs VRAM and compositing time. Promote **only** frequently-animated elements; add `will-change` *just before* the animation and **remove it after** (a permanent `will-change` on many nodes regresses memory and can *hurt* perf). Never blanket it across a component tree.
- **Threading:** main thread (JS / Style / Layout / Paint / `requestAnimationFrame`) vs **compositor thread** (scroll, and compositor-driven animations). **CSS + WAAPI animations of `transform`/`opacity` run off the main thread** → they stay smooth even while JS is busy. This is the single biggest reason to prefer CSS/WAAPI over JS rAF tweening; Motion's hybrid engine and GSAP both lean on the compositor where they can, but a JS `requestAnimationFrame` tween is main-thread-bound by definition.
- **Loading & asset hygiene:**
  - Images: `loading="lazy"` for below-fold; **never lazy-load the LCP image** — make it eager + `fetchpriority="high"` (and `<link rel="preload">` it if it's a CSS-background or late-discovered image). Serve **AVIF → WebP → fallback** via `<picture>`/`srcset`/`sizes`; always set `width`/`height` (or `aspect-ratio`) to reserve space and kill CLS.
  - JS: code-split + dynamic `import()` + tree-shake; rAF-throttle `scroll`/`resize`/`mousemove` and add `{ passive: true }` to scroll/touch listeners (lets the compositor scroll without waiting on your handler).
  - Fonts: `font-display: optional` for body (zero layout shift — uses fallback if the web font isn't cached within ~100ms, swaps only on next navigation) **or** `swap` + a **metric-matched fallback** (`size-adjust`, `ascent-override`) to minimize the swap reflow; `<link rel="preload" as="font" crossorigin>` the critical face; subset to the glyphs you ship. Variable fonts cut request count (one file, many weights).
  - Delivery: HTTP/2-3, Brotli, CDN; `content-visibility: auto` + `contain-intrinsic-size` to skip rendering offscreen sections (**Baseline since Sep 2025** — Chrome 85+, Firefox 125+, Safari 18+).
- **Core Web Vitals (field, p75, mobile + desktop separately):**
  - **LCP ≤ 2.5s** (≤4s = "needs improvement", >4s "poor"). Wins: fast TTFB, no render-blocking CSS/JS, preload/eager the LCP resource, avoid client-only rendering of the hero. *(The "good" bar is still 2.5s as of mid-2026 — the circulating claim of a 2.0s tightening in a "March 2026 core update" is SEO folklore, not in web.dev/Google docs.)*
  - **CLS ≤ 0.1.** Wins: dimension every image/video/iframe/ad/embed, reserve dynamic-content space, `font-display: optional`, **animate only `transform`/`opacity`** (never `top`/`height` for entrances), put toasts/banners in space that's already reserved or overlay them.
  - **INP ≤ 200ms** (replaced FID Mar 2024; **still the most-failed CWV** — ~86% of origins have "good" INP per the May 2026 CrUX, yet ~43% of *sites* still fail the 200ms threshold in broader scans, so treat it as the one most likely to bite). LCP and INP both went **Baseline (all major browsers) Dec 2025** so RUM now collects them everywhere, not just Chrome. Wins: break long tasks (>50ms) so input can be serviced; **`await scheduler.yield()`** between chunks to continue after yielding to higher-priority work — **Chrome + Firefox 142 (since Aug 2025), still NOT Safari → not Baseline**, so feature-detect and **keep the `setTimeout(...,0)` fallback** (or a timing-based "yield every ~50ms") so it degrades; yield *after* painting the visible response so the user sees feedback first; avoid layout thrashing in handlers; shrink DOM size; move heavy work to a Web Worker; prefer compositor animations so interaction handlers aren't competing with rAF.
  - **Tooling:** Lighthouse / PageSpeed Insights = **lab** (synthetic, one device) — good for catching regressions, *not* for pass/fail. **CrUX / RUM = field** (real users, p75) — what Search and your users actually experience; **decide ship/no-ship on field data.** Web Vitals JS lib or the `web-vitals` attribution build for in-page RUM; PerformanceObserver `long-animation-frame` (LoAF) to find the script that caused a slow INP.

```js
// Yield-aware long-task splitter (INP-safe). Paint the response, THEN yield.
async function runChunks(items, doWork) {
  let last = performance.now();
  for (const item of items) {
    doWork(item);
    if (performance.now() - last > 50) {           // every ~50ms…
      await (scheduler.yield?.() ??                  // Chrome/FF 142+ — NOT Safari
             new Promise(r => setTimeout(r, 0)));    // Safari fallback
      last = performance.now();
    }
  }
}
```

**Accessibility (ship conformance to WCAG 2.2 AA):**

- **Reduced motion — opt IN, don't just disable.** Author the *static* experience as the default and add motion only inside `@media (prefers-reduced-motion: no-preference)`; gate JS the same way (`matchMedia('(prefers-reduced-motion: reduce)')` + `.addEventListener('change', …)`; in GSAP use `matchMedia()`; don't init Lenis/ScrollSmoother under reduce). **Reduce, don't blanket-nuke**: drop parallax / scroll-jacking / autoplay loops / large slides / overshoot / spin; **keep** opacity fades and short, small moves so the UI still communicates state. (WCAG 2.3.3 Animation from Interactions, AAA, but treat as table stakes.)
- **The other accessibility media features** (production-relevant, all widely supported): **`prefers-reduced-transparency`** — drop `backdrop-filter`/translucency for solid fills (the correct fix for glassmorphism a11y); **`prefers-contrast: more`** — thicken borders, deepen text, remove low-contrast greys; **`forced-colors: active`** (Windows High Contrast Mode, ~93% reach) — don't fight the user's palette: use `system-color` keywords, keep `outline` on focus, mark meaningful images so they aren't dropped, test that icon-only buttons survive; **`prefers-color-scheme`** for dark mode.
- **WCAG 2.2 (Dec 2024 update; still the current Recommendation and the operative legal benchmark — ADA Title II, Section 508, EU Accessibility Act — in mid-2026).** WCAG 3.0 is a **March-2026 Working Draft** (174 "requirements", renamed from "outcomes"; expected Recommendation ~2028–2030) and **APCA is not a normative conformance target** in it. 2.2 added 9 SC — the load-bearing AA ones for UI:
  - **Target Size (Minimum) 2.5.8 (AA):** interactive targets ≥ **24×24 CSS px**, *or* spaced so a 24px circle centered on each doesn't overlap a neighbor. Exceptions: inline links in text, UA-controlled size, equivalent control elsewhere, or essential. (Touch ergonomics still favor ~44px; 24 is the legal floor, not the design target.)
  - **Focus Not Obscured (Minimum) 2.4.11 (AA):** the focused element must not be *entirely* hidden by sticky headers/footers/cookie bars — use `scroll-margin`/`scroll-padding` so it scrolls into view.
  - **Dragging Movements 2.5.7 (AA):** any drag interaction (sliders, reorder, kanban, map pan) needs a **single-pointer (tap/click) alternative**.
  - **Accessible Authentication (Minimum) 3.3.8 (AA):** no cognitive-test gate (no "retype this", no puzzle) — allow paste and password managers on auth fields.
- **Semantic HTML first (first rule of ARIA: don't use ARIA if HTML can do it):** native `<button>`, `<a href>`, `<nav>`, `<main>`, `<dialog>`, `<details name>` (Baseline Sep 2025 — exclusive accordion with zero JS) over `<div role>`. One `<h1>`, logical non-skipping heading order, landmark regions, an accessible name on every interactive element (icon-only buttons need `aria-label`/visually-hidden text). Don't override implicit roles.
- **Focus & keyboard:** never bare `outline: none`. Style **`:focus-visible`** with a thick (≥2px), high-contrast ring that meets 3:1 against adjacent colors (consider `outline-offset` for separation). Logical DOM/tab order, **no positive `tabindex`**, a skip link as the first focusable element. Modals: prefer native `<dialog>` + `showModal()` (gives focus-move-in, top-layer, backdrop, `Esc`-to-close for free) and add **`closedby="any"`** for light-dismiss (Chrome/Edge/Firefox now; Safari catching up via Interop 2026 — keep a JS backdrop-click fallback). If you hand-roll a modal: move focus in, **trap** it, `Esc` closes, **restore focus to the trigger** on close, and mark background `inert`. (For a no-JS path, native top-layer overlays now come from the **Popover API + Invoker Commands** — see §9.)
- **Color & non-color cues:** text **≥ 4.5:1** (≥3:1 for ≥24px or ≥18.66px-bold), UI components / icons / focus indicators / chart strokes **≥ 3:1**; **never encode meaning by color alone** (add text, icon, or pattern). Verify with WebAIM or the DevTools contrast picker.
- **Screen readers & async:** correct `alt` (and `alt=""` for purely decorative); announce async changes with `aria-live="polite"` / `role="status"` (or `assertive`/`role="alert"` for errors); name form fields with real `<label>`s and wire errors via `aria-describedby` + `aria-invalid`. **Canvas/WebGL/SVG-art is opaque to AT** → provide a DOM/table/text equivalent and don't put essential content or navigation only in the canvas.
- **Motion-specific WCAG:** auto-playing/looping motion that runs **>5s** needs a pause/stop/hide control (2.2.2); content must **not flash >3×/sec** (seizure risk, 2.3.1/AA, 2.3.2/AAA); gate parallax and scroll-jacking on reduced motion (above).

## §15. Expensive vs. cheap (the cross-cutting truth)

**Reads as expensive:** Swiss-grid restraint; generous *active* whitespace (negative space used as composition, not just padding); a distinctive non-default typeface with hard scale contrast and tight display tracking; a whisper of grain to kill banding; perceptual (OKLCH/LCH) dark themes with **elevation-by-lightness** (translucent-white overlays, not drop shadows); one disciplined low-saturation accent following 60-30-10; a real grid + baseline rhythm on an 8-pt scale; purposeful, sparse, compositor-driven motion; refraction/liquid/3D as a *single earned moment*, lazy-loaded with a static fallback.

**Reads as cheap (AI-slop tells):** pure `#000` background + low-contrast grey body text; default Inter/Playfair/Roboto with no scale and no tracking; **glassmorphism on everything**; the purple→blue gradient + frosted card combo (the #1 vibe-coded tell); uniform-size bento boxes (kills hierarchy); two-stop "mesh" gradients; neon glow faked as one blurry `box-shadow`; constant glitch/warp loops; neumorphism (fails 3:1 by construction); visible tiled-PNG noise that repeats; centered-everything with no asymmetry; `outline: none` with no `:focus-visible` replacement; layout-shifting hero; entrance animations on `top`/`height`.

**Always-elevates techniques (any aesthetic):** (1) grain via SVG `feTurbulence` at opacity .03–.08 + `mix-blend-mode: overlay`; (2) generous active whitespace — start with too much, then remove; (3) **restraint** — one accent, one hero moment, one technique done correctly; (4) a typeface with a point of view (real optical sizes + figure styles + small caps); (5) perceptual color on a real token system (primitive → semantic → component); (6) **execute the system, not the surface** — a real grid, a real light source, a real type scale, a real color system underneath the style; (7) honor the OS — `prefers-reduced-motion`, `prefers-color-scheme`, `prefers-contrast`, `forced-colors` (respecting user intent *is* a premium signal, and the inverse — ignoring it — is a tell).

**The two worlds — know which you're building for *before* you start:**
- **Awards / portfolio:** maximal WebGL, kinetic type, scroll choreography, custom cursors, smooth-scroll. Motion *is* the product; you accept the perf/a11y cost and pay it down deliberately (lazy-load, static fallbacks, reduced-motion paths). Reference: Awwwards SOTY-tier sites — the Webflow-shell + WebGL + Rive + Lenis stack.
- **Production / business:** bento grids, restraint, real content, fast field CWV, full keyboard/SR accessibility, sub-200ms INP. The flashy award trends frequently **don't ship** here — and shouldn't. "Expensive" in this world means *systematized and fast*, not maximal.

The mistake either way is borrowing one world's tactics without its discipline: award flash on a content site that then fails LCP/INP, or production restraint so timid it reads as a template. Pick the world, then commit to its full cost.

## §16. Pre-ship gates (a beautiful site isn't done until it passes these)

**Performance (field, p75, decide on CrUX/RUM — not lab Lighthouse):**
- LCP ≤ 2.5s, CLS ≤ 0.1, INP ≤ 200ms.
- Animate **only `transform`/`opacity`** (and compositor-safe `filter`/`clip-path` for fade/move, not radius animation).
- Reads batched before writes (no forced sync reflow in handlers/animation loops).
- `scroll`/`resize`/`mousemove` rAF-throttled and `{ passive: true }`.
- No long tasks > 50ms — split with `scheduler.yield()` (Chrome + Firefox 142, **not Safari** → keep the `setTimeout`/timing-based fallback) and yield *after* painting the visible response.
- `will-change` applied sparingly and removed after the animation (not left on the tree).
- LCP image eager + `fetchpriority="high"` (preloaded if late-discovered); below-fold media `loading="lazy"`.
- AVIF/WebP via `<picture>`/`srcset`; **every** image/video/iframe has explicit dimensions or `aspect-ratio`.
- Fonts: preload + subset + `font-display: optional` (or `swap` with a metric-matched fallback).
- Heavy animation is CSS/WAAPI/compositor-driven; offscreen sections use `content-visibility: auto`.
- Third-party scripts deferred/audited (a common INP and LCP regressor); verified on a mid-tier mobile device, not just desktop.

**Accessibility (WCAG 2.2 AA):**
- `prefers-reduced-motion` honored in **CSS *and* JS**, opt-in pattern (static is the default; Lenis/ScrollSmoother not initialized under reduce).
- `prefers-reduced-transparency` / `prefers-contrast` / `forced-colors` (Windows HCM) don't break the layout — glass falls back to solid, focus rings survive, meaningful icons aren't dropped.
- No content flashes > 3×/sec; auto-motion running > 5s has a pause/stop/hide control.
- Semantic HTML + landmark regions + one `<h1>` + logical, non-skipping heading order.
- Visible **`:focus-visible`** indicator everywhere (≥2px, ≥3:1 contrast); **no bare `outline: none`**; no positive `tabindex`; skip link first.
- Modals: native `<dialog>` + `showModal()` (+ `closedby="any"` where supported, JS backdrop fallback) **or** hand-rolled with focus move-in + **trap** + `Esc` + focus-restore + background `inert`; native overlays/popovers via the Popover API + Invoker Commands get top-layer/focus/Esc for free.
- Interactive targets ≥ 24×24 CSS px (or adequately spaced); inline-text-link exception understood.
- Focused element never fully obscured by sticky chrome (use `scroll-margin`/`scroll-padding`).
- Every drag interaction has a single-pointer (tap/click) alternative.
- Auth fields allow paste + password managers (no cognitive-test gate).
- Contrast: text ≥ 4.5:1 (≥3:1 large), UI/icons/focus ≥ 3:1; **never color-alone** for meaning.
- Correct `alt` (decorative = `alt=""`); async updates announced via `aria-live`/`role="status"`/`role="alert"`.
- Canvas/WebGL/SVG-art has a DOM/text equivalent; essential content & nav are not canvas-only.
- **Tested keyboard-only, with a screen reader (NVDA/VoiceOver), and with an automated scan (axe / Lighthouse a11y)** — automated catches ~30–40% of issues; the manual passes catch the rest.

## How to use this knowledge

Foundations first (Refactoring UI, type scale, OKLCH color, whitespace, Gestalt, UX laws) — they make *everything* better. Then layer the flashy tools (GSAP, Three.js, shadcn) only after the system underneath is sound. Use §3's named aesthetics as search vocabulary, §11's galleries to reverse-engineer the masters, §15 as the taste filter, and §16 as the non-negotiable gate. Always decide which of the two worlds (§15) you're building for before you start.
